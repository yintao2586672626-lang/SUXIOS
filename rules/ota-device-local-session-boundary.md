# 宿析OS OTA 设备本地登录状态与门店数据执行边界

## 1. 规则目的

宿析OS的业务服务器可以集中保存结构化经营数据，但不得集中承载所有门店的浏览器登录状态。谁在自己的电脑登录携程、美团或其他 OTA，浏览器会话就归谁的设备管理；该用户只在授权门店范围内执行采集并向服务器提交结果。

本规则适用于浏览器 Profile、Cookie/API 辅助、Python 自动化、Node/Playwright/CDP、插件、书签脚本、设备绑定、任务派发、采集结果上传和临时文件管理。

## 2. 系统角色

| 角色 | 允许保存和执行的内容 |
|---|---|
| 宿析OS业务服务器 | 用户与酒店权限、设备登记、平台账号别名、门店映射、任务状态、结构化字段事实、最小来源证据、失败原因、分析与运营结果 |
| 用户采集设备 | 本机浏览器 Profile、Cookie、localStorage、sessionStorage、验证码/二次验证状态、本机自动化运行环境和短期临时文件 |
| OTA平台 | 平台账号、平台门店身份和授权后台数据的原始来源 |

服务器部署在某位用户电脑上时，该电脑可以同时扮演业务服务器和该用户自己的采集设备，但两个职责必须逻辑分开；这不代表服务器可以接管其他用户的登录状态。

## 3. 强制边界

1. 浏览器登录状态只在实际登录设备上保存和使用，不上传 Profile 目录、完整 Cookie、长会话令牌、Authorization 或验证码状态到业务服务器。
2. 服务器不得建立集中 Profile 池，不得要求所有门店账号统一在服务器电脑登录，也不得自动复制其他用户设备的浏览器状态。
3. 用户设备只能执行当前登录用户有权访问的酒店。任务领取、结果提交和数据库写入都必须再次校验租户、用户、系统酒店、平台和平台门店身份。
4. 同一平台账号管理多家酒店时，按“设备 + 平台 + 平台账号”复用一个本地 Profile，再映射多个已授权 `system_hotel_id`；除非平台隔离要求有证据，不按酒店复制 Profile。
5. 默认采用手动触发或用户确认后的半自动执行。定时任务默认关闭；设备离线、未登录、验证码阻塞或授权失效时保持 `device_offline`、`login_required`、`verification_required`、`permission_denied` 或 `collection_failed` 等真实状态。
6. 任务不得静默转移到服务器或另一位用户设备执行。需要换设备时，必须由有权用户在新设备重新登录并重新建立设备绑定。
7. Python、Node、插件和浏览器脚本属于设备侧采集适配器，不能绕过服务器的保存、回读、幂等、数据质量和酒店权限校验。

## 4. 服务器允许保存的设备绑定信息

服务器可以保存以下非会话信息：

- `tenant_id`
- `user_id`
- `collector_device_id`（非敏感、可轮换的设备标识）
- `platform`
- 平台账号别名或脱敏标识
- 授权的 `system_hotel_id` 与平台门店标识映射
- 采集器版本、能力和在线状态
- `last_seen_at`、`last_success_at`、最近失败阶段与失败原因
- 任务ID、目标日期、采集方式、质量状态、保存数量与回读结果

上述信息不得包含可直接恢复登录状态的 Cookie、令牌、密钥、Profile 压缩包或敏感请求头。

## 5. 数据上传与本地存储规则

1. 用户设备优先上传标准化字段和最小必要来源证据，不默认上传完整网页、敏感 HAR、批量截图或无边界的原始响应。
2. 每次上传必须包含 `platform`、`tenant_id`、`user_id`、`collector_device_id`、`system_hotel_id`、平台门店标识、`data_date`、`collected_at`、`source_method` 和质量状态。
3. 本地临时文件必须集中到受控目录，并配置单任务大小、总容量、文件数量和保留时间上限；确认服务器保存和回读成功后，允许安全清理。
4. 采集失败不得上传旧数据、其他门店数据、空数组或默认值冒充成功。历史结果只能明确标记为历史存在。
5. 服务器只把通过身份、来源、日期、字段和回读校验的数据用于收益分析、AI建议和运营复盘。

## 6. 手动与自动化规则

| 模式 | 默认状态 | 规则 |
|---|---|---|
| 手动采集 | 默认启用 | 用户选择平台、酒店、日期和数据类型后执行 |
| 半自动采集 | 按需启用 | 用户确认后由本机 Python、Node 或浏览器辅助工具执行重复步骤 |
| 定时采集 | 默认关闭 | 仅对已经验证稳定的账号、设备和字段单独开启，必须可暂停 |
| 服务器集中代采 | 禁止作为正式方案 | 仅兼容服务器拥有者本人的单用户本机模式，不扩展为多用户Profile池 |

自动化必须设置超时、有限重试、可取消状态和清理策略。验证码、二次验证、登录失效或平台风控出现时立即停止并交还用户处理。

## 7. 兼容与例外

1. 当前已存在的服务器本机 Profile 可以继续服务于服务器拥有者本人，但必须标记为 `single_user_local`，不得默认绑定其他用户或批量扩展到所有门店。
2. 用户明确配置的、与浏览器登录状态无关的后台 API 凭据，可以按现有凭据保险库规则加密保存；这不构成上传浏览器 Profile 或会话状态的授权。
3. 任何例外都必须说明用户、设备、平台、门店范围、数据用途、保存位置和撤销方式，不得使用全局默认例外。

## 8. 验收检查

涉及登录、设备绑定或采集派发的功能，至少验证：

1. 两位用户分别登录后，双方 Profile 和 Cookie 只存在各自设备。
2. 服务器数据库和文件目录不存在可恢复用户浏览器会话的 Profile 或完整 Cookie。
3. 用户只能领取和提交其授权酒店的任务；越权结果被拒绝且不写库。
4. 一个账号管理多家酒店时可以复用账号级 Profile，同时每条结果仍准确绑定目标门店。
5. 设备离线或登录失效时显示真实状态，不自动由服务器或其他用户代采。
6. 采集完成后服务器保存数量、关键字段、来源、日期和页面回显一致。
7. 本地临时文件达到容量或保留期上限时可以安全清理，不影响已入库事实。

## 9. 禁止事项

- 禁止把几百家门店的浏览器 Profile 集中保存在宿析OS服务器。
- 禁止跨用户复制、共享或复用浏览器会话。
- 禁止把完整 Cookie、长令牌或 Profile 包装成普通业务字段上传。
- 禁止设备离线后静默改由服务器或其他员工设备执行。
- 禁止未经酒店权限校验直接接收设备上报结果。
- 禁止用自动化便利性绕过人工确认、验证码、平台权限或真实失败状态。
