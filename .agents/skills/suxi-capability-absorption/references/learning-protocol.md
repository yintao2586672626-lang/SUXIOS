# 学习强化协议

## 1. 自动选择学习方法

| 方法 | 适用材料 | 必须取得的证据 |
| --- | --- | --- |
| 白盒学习 | 源码、Skill、配置、API规范、数据库结构、工作流JSON/YAML | 真实入口、依赖、数据合同、错误路径 |
| 黑盒学习 | 网站、应用、桌面软件、外部服务 | 可见输入、操作、输出、状态和失败行为 |
| 示教学习 | 录屏、屏幕演示、语音讲解、人工步骤 | 动作顺序、操作者意图、关键判断和完成证据 |
| 样例学习 | 报表、模板、截图、输入输出对、历史成品 | 不变结构、可变字段、质量标准和反例 |
| 失败学习 | 日志、报错、测试、工单、差评、用户纠错 | 触发条件、原始失败、正确结果和回归样例 |
| 差异学习 | Commit、Diff、Changelog、新旧版本 | 变化点、兼容影响、迁移规则和复验范围 |
| 组合学习 | 同时提供网站、源码、文档、数据或示教 | 来源间一致项、冲突项、证据优先级和未知项 |

优先选择能直接证明行为的方法。重要数据链或高影响写入不能只靠截图、宣传文案或单个成功样例。

## 2. 能力合同

在实现前整理最小合同，不要求用户手工填写：

```text
来源标识 / 版本或日期 / 授权与数据范围 / 目标使用者 /
触发条件 / 输入 / 核心动作 / 输出 / 状态变化 / 副作用 /
正常样例 / 边界样例 / 失败样例 / 允许差异 / 未知项 / 验收命令
```

- 运行中的真实行为用于说明“当前是什么”；用户确认的业务目标用于说明“应该是什么”，两者冲突时不能互相覆盖。
- OTA页面或渠道数据只证明渠道范围，不扩大为全酒店事实。
- 来源无法运行时保留确切失败证据，然后切换适配、行为复刻或最小重建。

## 3. 黄金样例与重放

1. 优先保存最小脱敏输入、预期输出、状态变化和错误结果。
2. 至少重放一个主路径；材料允许时增加一个边界和一个失败样例。
3. 对页面比较任务完成与状态，不只比较像素；对API比较字段、状态码和错误；对SOP比较状态流转和证据；对数据比较语义、范围、保存和回读。
4. 原材料缺少失败样例时标记 `unknown`，不得用自造错误假装来源已经验证。
5. 用户纠正结果后，把纠正样例加入 eval 或 verifier，并同时复跑原始样例，防止修一处坏一处。
6. 交付模式必须同时验证一条代表性成功路径和一条关键失败、缺失或未验证路径；后者可以验证宿析OS自己的保护行为，但不得冒充来源系统的真实失败行为。

## 4. 多源冲突与置信边界

按以下顺序处理证据，而不是简单投票：

1. 当前可重复运行或可观察的真实行为；
2. 当前自动化测试、接口规范和数据约束；
3. 经用户确认的业务规则、SOP和目标结果；
4. 当前文档、示例成品和版本说明；
5. 推断。

来源冲突时分别记录“来源现状”和“目标决策”。缺少决定性证据时使用 `partial` 或 `blocked`，不把推断写成事实。

## 5. 防遗忘与升级

功能闭环后按实际需要沉淀，不为沉淀而扩大任务：

- 来源指纹：路径、仓库提交、版本、接口规范版本、页面日期或文件哈希；
- 稳定合同：字段映射、状态机、错误语义、允许差异和权限边界；
- 可重放证据：脱敏 fixture、黄金样例、聚焦测试或 verifier；
- 自动触发：重复任务写入项目 Skill，重复失败写入 `evals/`；
- 纠错记录：用户明确纠正的稳定规则进入合同和回归样例；
- 定向升级：来源指纹变化时只重验受影响合同，不从头重学全部材料。

完成报告必须说清：当前阶段、真实验证、未验证项、来源版本和下一次升级触发条件。

## 6. 数据真实性放行门

外部材料一旦进入 `OTA数据 -> 收益分析 -> AI决策` 链路，逐项检查：

```text
平台 / 系统酒店 / 平台门店或绑定 / 目标日期 /
来源方法 / 采集或导出时间 / 质量状态 / 持久化与数据库回读
```

- 缺少关键身份或日期时保持 `binding_missing`、`unverified`、`stale`、`partial` 或对应失败状态。
- 文件能解析、接口返回200、页面显示数字，都不能单独证明数据可用于收益或AI决策。
- OTA渠道证据只进入渠道分析，不提升为全酒店经营事实。

## 7. Skill质量基线

1. 把触发准确性与输出质量分开测试：`evals/trigger-evals.json`覆盖应触发和近似但不应触发的语境；`evals/evals.json`保存真实任务、预期结果和可核对断言。
2. 触发测试保持正负样例接近平衡，混合正式、口语、错别字、短句、长上下文和关键词近似场景。
3. 输出质量测试在干净上下文中分别运行新版Skill与旧版或无Skill基线，避免开发对话、先前产物和预期答案泄漏。
4. 机械结果用脚本断言，复杂体验保留人工复核；PASS必须引用具体输出证据，不给模糊结果“酌情通过”。
5. 条件允许时记录用时和token成本；新版只增加复杂度却没有明显提高通过率时，删减规则而不是继续堆说明。
6. 阅读执行轨迹：若智能体反复扫描、绕路或忽略步骤，优先消除模糊、无关或互相冲突的指令。
7. 迭代到失败断言得到修复、人工反馈稳定为空，或新版本相对基线不再有实质改进后停止。
8. 触发评测与现有Skill共存运行，记录实际加载路径、客户端和文件树指纹；避免宽泛description抢占别的能力。
9. 升级先形成候选版本，验证优于旧版后再切换；高影响能力保留上一通过版本作为回滚点。
