# 老板视角信息架构优化清单

> 状态：只读体检后的优化依据。
> 范围：统一展示界面问题、入口重复、后续收敛方向。
> 本文不改 UI、不改路由、不改数据库、不改 OTA 采集逻辑。

## 1. 用户判断

当前系统的主要使用者更接近老板或经营负责人，不是每天长时间操作系统的门店员工。

老板默认关心：

1. 今天有没有异常。
2. 钱从哪里来、哪里少了。
3. OTA 数据是否可信。
4. AI 建议能不能执行。
5. 谁负责处理，处理后有没有效果。
6. 是否影响后续投资、开店、扩张或转让判断。

因此界面优化不应继续把所有运营细节平铺出来，而应先把问题统一展示为：

```text
结论 -> 风险 -> 金额影响 -> 证据状态 -> 建议动作 -> 负责人/进度 -> 投决影响
```

## 2. 主链路不变

产品链路仍然保持：

```text
OTA 数据 -> 收益分析 -> AI 决策 -> 运营管理 -> 投资决策
```

关键边界：

- OTA 数据只代表 OTA 渠道口径，不自动等于全酒店经营口径。
- 采集失败、登录失效、缺字段、缺表、fallback-only 必须直接展示。
- AI 建议默认是待确认建议，不能表现成已自动执行。
- 投决判断必须看到证据来源和经营复盘，不应只看单次 OTA 数据。

## 3. 当前主要问题

| 问题 | 表现 | 对老板的影响 | 优化方向 |
|---|---|---|---|
| 首页信息过密 | 今日结论、快捷入口、闭环步骤、天气、趋势、竞对、执行摘要同时出现 | 不容易快速判断“今天要不要管” | 首页只保留结论、风险、动作、钱和证据状态 |
| 入口重复 | `AI工作台`、`经营概览`、首页快捷入口、闭环步骤都能进入相近页面 | 老板不知道从哪里开始 | 统一为“今日老板看板/今日经营看板” |
| OTA 数据页过重 | 数据健康、平台账号、自动获取、记录、字段、下载、配置混在一起 | 数据可信度和配置诊断混淆 | 主入口只显示“数据是否可信”，配置/日志/字段下沉 |
| 收益能力分散 | 收益分析中心、Revenue AI、Agent 收益管理存在认知重叠 | 不清楚哪个是最终收益判断 | 收敛成“收益诊断与调价建议” |
| 运营执行拆得过细 | 根因、预警、日报、策略、执行分散为多个页面 | 适合运营岗，不适合老板快速看 | 老板只看“待处理事项/执行进度/复盘结果” |
| 低频模块占入口心智 | 筹建、开业、扩张、转让、图片、工具箱仍容易被当成主功能 | 稀释日常经营主链路 | 统一归为“投资与资产辅助” |
| 配置和诊断散落 | 系统配置、数据配置、平台账号、Cookie/API、下载、日志分散 | 问题排查成本高 | 建立“配置与证据中心”概念 |

## 4. 建议的统一展示模型

后续即使暂时不改界面，也应按以下模型统一文案、文档、验收和后续 UI 设计。

| 层级 | 老板要看的问题 | 应统一展示 | 细节下沉到 |
|---|---|---|---|
| L1 今日结论 | 今天经营是否正常 | 收入、订单、转化、风险、下一步动作 | 经营明细 |
| L2 钱的影响 | 少赚/多赚在哪里 | OTA 收入、间夜、ADR、竞对价差、转化损失 | 收益分析明细 |
| L3 数据可信度 | 这个结论可靠吗 | 平台登录、采集状态、字段缺口、数据日期 | 数据健康/配置诊断 |
| L4 AI 建议 | 建议能不能做 | 建议动作、阻断原因、证据来源、人工确认状态 | AI 日报/治理日志 |
| L5 执行状态 | 谁在处理，效果如何 | 负责人、进度、证据、ROI 复盘 | 执行跟踪 |
| L6 投决影响 | 是否影响投资判断 | 现金流、收益质量、风险等级、扩张/转让影响 | 投资与资产辅助 |

## 5. 页面收敛建议

| 当前入口/页面 | 建议定位 | 收敛方式 |
|---|---|---|
| `AI工作台` + `经营概览` | 今日老板看板 | 合并认知，不再两个入口表达同一首页能力 |
| `online-data` | 数据可信度工作台 | 默认只看数据健康；平台自动获取、账号绑定、字段配置下沉 |
| `ctrip-ebooking` | 携程 OTA 证据页 | 保留平台专页，但避免和数据健康重复解释状态 |
| `meituan-ebooking` | 美团 OTA 证据页 | 保留平台专页，默认聚焦榜单/流量/订单证据 |
| `revenue-research-center` | 收益诊断 | 作为钱和调价判断主入口 |
| `agent-center` | 高级 AI 工具箱 | 从老板日常路径移除，作为管理员/高级工具 |
| `ai-daily-report` | AI 建议归档与转执行 | 负责建议、阻断、转执行，不做首页替代 |
| `ops-track` | 执行与复盘 | 老板重点看状态和 ROI，不看所有操作明细 |
| `lifecycle` 及筹建/开业/扩张/转让页面 | 投资与资产辅助 | 低频但保留，作为投决沉淀入口 |
| `system-config` / `data-config` / `operation-logs` | 配置与证据中心 | 管理员入口，不与经营主链路同级 |

## 6. 先不动界面时可以做的优化

不改 UI 的前提下，可以先做这些低风险优化：

1. 统一文档口径：确认后续所有文档都按老板视角描述入口。
2. 统一命名口径：把“收益分析中心 / Revenue AI / 收益管理 Agent”归到同一个能力说明。
3. 建立入口治理表：每个入口标注高频/低频、角色、数据来源、是否主链路。
4. 建立问题展示标准：异常必须按“问题、影响、证据、动作、负责人、复盘”展示。
5. 建立状态标准：分开展示“已登录、可采集、有数据、可决策、可执行”。
6. 建立低频入口清单：生命周期、投决、工具箱、图片优化默认归入辅助区。

## 7. 入口治理表

后续优化入口时，先按这张表判断是否应该留在主链路。没有进入主链路的入口不删除，只下沉。

| 入口类型 | 老板是否高频 | 默认位置 | 展示原则 | 不应做的事 |
|---|---:|---|---|---|
| 今日经营结论 | 是 | 首页第一屏 | 一句话结论、风险、金额影响、下一步动作 | 不展示完整运营明细 |
| OTA 数据可信度 | 是 | 首页摘要 + OTA 数据页主入口 | 登录、采集、字段、数据日期分开展示 | 不用一个绿灯代表全部正常 |
| 收益诊断 | 是 | 主链路 | 收入、间夜、ADR、转化、价差和调价前置条件 | 不把 OTA 收益包装成全酒店收益 |
| AI 建议 | 是 | 主链路 | 待确认、阻断原因、证据来源、可转执行状态 | 不表现为自动执行完成 |
| 执行跟踪 | 是 | 主链路 | 负责人、进度、证据、ROI 复盘 | 不把未复盘动作算成已闭环 |
| 投决沉淀 | 中 | 主链路末端摘要 | 只展示经营复盘对投资判断的影响 | 不从单日 OTA 数据直接推投决结论 |
| 平台配置 | 否 | 配置与证据中心 | 只在诊断、管理员、缺口处理时出现 | 不和经营结果同级展示 |
| 下载/日志 | 否 | 配置与证据中心 | 作为排障证据 | 不占老板首页或主导航心智 |
| 字段配置 | 否 | 数据治理/管理员入口 | 标明字段状态和验证结果 | 不让老板承担字段维护任务 |
| 生命周期辅助 | 低 | 投资与资产辅助 | 筹建、开业、扩张、转让统一归档 | 不挤占日常经营入口 |
| AI/图片/高级工具箱 | 低 | 更多功能/高级工具 | 明确是工具，不是经营主线 | 不与收益分析、执行跟踪重复命名 |

## 8. 状态展示标准

界面后续优化时，所有状态都按下面拆开，不再合并成模糊成功态。

| 状态层 | 含义 | 可显示为正常的条件 | 必须直接暴露的异常 |
|---|---|---|---|
| 系统登录 | 用户能否进入宿析OS | Token 有效、权限匹配 | 未登录、无权限、角色不匹配 |
| 平台绑定 | 门店是否绑定携程/美团账号或数据源 | 门店和平台身份明确 | 未绑定、门店不匹配、平台身份不清 |
| 平台登录态 | 是否能复用 OTA 后台会话 | Profile/Cookie/API 辅助可用且未过期 | 短信/验证码、登录失效、Profile 缺失 |
| 采集执行 | 是否成功发起采集 | 任务已执行并返回结构化响应 | 采集失败、接口未命中、平台拦截 |
| 字段证据 | 关键字段是否命中 | source path、metric key、storage field 可追溯 | 字段缺失、字段语义不明、样例不足 |
| 数据可用 | 目标日期是否有可用业务数据 | 有目标日期数据、非空、非过期 | missing_table、missing_data、stale_data |
| 决策可用 | 是否能支撑 AI/规则建议 | 关键输入齐全且边界清楚 | fallback-only、blocked、unverified |
| 执行可用 | 是否能转为执行动作 | 人工确认、执行入口、负责人和证据要求明确 | execution_blocked、缺负责人、缺复盘口径 |

## 9. 小补丁执行清单

为了不动大页面，后续按“小补丁优先”执行：

| 顺序 | 优化项 | 首选改动位置 | 是否触碰大页面 | 验收方式 |
|---:|---|---|---|---|
| 1 | 统一老板视角命名 | 文档、验收说明、入口治理表 | 否 | 所有主链路文案指向同一模型 |
| 2 | 固定入口分级 | 文档或轻量配置说明 | 否 | 每个入口都有高频/低频和角色归属 |
| 3 | 固定状态口径 | 文档、测试说明、后续 helper 约束 | 否 | “登录/采集/数据/决策/执行”不混成一个状态 |
| 4 | 收敛低频模块说明 | 文档、后续菜单策略 | 否 | 生命周期和工具箱明确为辅助 |
| 5 | 首页减噪方案 | 设计说明先行 | 暂不触碰 | 明确哪些模块保留、折叠、下沉 |
| 6 | OTA 页重排方案 | 设计说明先行 | 暂不触碰 | 数据健康优先，配置诊断后置 |

## 10. 第二层优化点

### 10.1 Frontend

| 优化点 | 建议 | 边界 |
|---|---|---|
| 首页信息密度 | 再降一档，只保留“今日结论 + 下一步动作 + 闭环进度” | 天气、趋势、竞对、辅助说明默认折叠 |
| OTA 页面任务顺序 | 按任务重排：先看“数据是否可信”，再看“携程/美团数据”，最后看配置、下载、日志、字段 | 不先重做视觉，不删除已有入口 |
| Agent 中心定位 | 改成“高级工具箱”，默认不进入日常路径 | 收益、资产、员工、日志不再和主链路竞争 |

### 10.2 Backend / Admin

| 优化点 | 建议 | 边界 |
|---|---|---|
| 菜单口径 | 把“可见 IA 配置”沉淀成单独结构 | 避免继续在大页面里临时重组菜单 |
| 旧 path 兼容 | 保留旧 path，统一入口文案 | “收益分析中心 / Revenue AI / 收益管理 Agent”归成一个用户认知 |
| 配置中心 | 系统配置、数据配置、平台账号绑定统一收口 | 不再散落在 OTA 页、门店页、系统页 |

### 10.3 Security

| 优化点 | 建议 | 边界 |
|---|---|---|
| 登录层分离 | 平台账号绑定、Cookie/API 辅助、Profile 登录态继续和系统登录分开展示 | 不把系统登录成功当成 OTA 可采集 |
| 高风险入口降曝 | 下载中心、日志、抓取设置减少默认曝光 | 只在诊断或管理员场景出现 |
| 状态分层 | “可采集 / 已登录 / 有数据”分开展示 | 不合并成一个绿色状态 |

### 10.4 Performance

| 优化点 | 建议 | 边界 |
|---|---|---|
| `online-data` 优先级 | 先拆 Tab 加载，再考虑视觉 | 它是重型界面，不能先做装饰性优化 |
| 首页快捷入口 | 保留一个统一跳转函数和一个统一加载调度 | 避免多组入口触发同类加载 |
| 低频模块预热 | 生命周期、Agent 工具箱、图片优化不参与首页预热 | 保留入口，但不拖慢老板首页 |

### 10.5 Other

| 优化点 | 建议 | 边界 |
|---|---|---|
| 文档口径同步 | 同步旧导航口径 | 防止后续 UI 优化反复回到旧结构 |
| 入口治理清单 | 每个入口标明高频/低频、角色、数据来源、是否主链路、是否只读 | 作为后续小补丁前置依据 |

### 10.6 最值得先做

| 优先级 | 改进 | 目标 |
|---:|---|---|
| 1 | 首页减噪 | 只保留今日经营主线 |
| 2 | OTA 数据页重排 | 健康状态优先，配置诊断后置 |
| 3 | 菜单口径统一 | 源码菜单、可见菜单、文档验收口径保持一致 |

### 10.7 可新增功能

| 功能 | 价值 | 边界 |
|---|---|---|
| 角色化首页 | 店长、运营、管理员看到不同默认入口 | 先定义角色入口表，再做 UI |
| 闭环状态总览 | 一屏显示 OTA 数据、收益分析、AI建议、执行、复盘、投决沉淀真实状态 | 只展示状态，不伪造闭环完成 |

## 11. 后续执行顺序

### P0：先统一认知，不动界面

1. 固定老板视角入口模型。
2. 统一主链路文案。
3. 建立入口治理表。
4. 建立状态展示标准。

### P1：轻量调整入口，不重做页面

1. 合并 `AI工作台` 与 `经营概览` 的表达。
2. OTA 默认入口聚焦数据可信度。
3. 配置、日志、下载、字段诊断降级为二级入口。

### P2：再做页面级 UI 优化

1. 首页减噪。
2. OTA 数据页重排。
3. 收益/AI/执行链路重新组织。
4. 投资与资产辅助模块统一归档。

## 12. 验收标准

后续优化完成时，应满足：

1. 老板进入系统 30 秒内能看到今天是否需要处理。
2. 每个异常都有金额/风险影响、证据状态和下一步动作。
3. OTA 渠道口径和全酒店口径不混用。
4. AI 建议不会表现成自动执行结果。
5. 配置、日志、下载、字段诊断不占据日常主入口。
6. 低频投资/筹建/扩张/转让能力保留，但不干扰日常经营判断。
7. 旧 path 和旧数据保留兼容。
