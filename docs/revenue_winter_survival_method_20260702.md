## 一、核心方法

### 1.1 双率驱动：出租率保现金流，市占率保未来

资料的主线是用“市占率”和“出租率”共同判断区域竞争力：

| 指标 | 方法含义 | 宿析OS使用边界 |
| --- | --- | --- |
| 出租率 | 实际售出间夜 / 可售间夜，反映门店现金流和经营效率 | 需要 PMS、房态或日报房量；OTA 订单不能直接等同全店出租率 |
| 市占率 | 本店客源或间夜 / 区域同类客源或间夜，反映抢客能力 | 需要明确商圈、竞对集合和分母；缺分母时只能输出“不可计算” |
| 区域竞争力 | 市占率与出租率共同决定的经营状态 | 可作为 AI 诊断框架，不应写成未验证的固定公式 |

经营顺序：

1. 寒冬期优先守住出租率和现金流底线。
2. 在不破坏底价和品牌的前提下，提高本店在商圈里的订单份额。
3. 复苏期再把高市占率转化为更高 RevPAR 和资产回报。

### 1.2 旺季与寒冬的策略切换

| 场景 | 优先目标 | 策略倾向 | 风险控制 |
| --- | --- | --- | --- |
| 旺季 | 利润最大化、ADR 优先 | 收紧低价渠道、控制库存、保护高价订单 | 防超售、防 No-show、控制客诉 |
| 寒冬 | 现金流不断裂、OCC 优先 | 动态让利、全渠道承接、本地客/会员/长单补底 | 防空转、防价格踩踏、防份额流失 |

宿析OS中需要把“旺季/寒冬策略”做成状态判断，而不是固定规则。判断依据应来自：Pace、Pickup、未来 14 天空窗、竞对价格、商圈流量、转化率、库存和底价约束。

## 二、可落地的经营闭环

### 2.1 数据诊断先行

每次给调价、促销或渠道建议前，先完成四类数据诊断：

| 诊断项 | 需要看的数据 | AI 可输出 |
| --- | --- | --- |
| Pace 进度 | 截止当前的累计预订量、目标基线、历史同期或近期均值 | 进度领先/正常/落后，是否需要干预 |
| Pickup 速度 | 最近 1/3/7 天新增订单或间夜 | 需求加速、放缓、异常突增 |
| 竞对位置 | 同商圈竞对价格、房型、活动、排名、可售状态 | 我方价格竞争力、是否被低价抢量 |
| 流量转化 | 曝光、浏览、咨询、订单、取消、支付、入住 | 判断问题在流量、转化、价格还是库存 |

缺少任一关键分母时，系统应输出数据缺口，不给确定性结论。

### 2.2 Pace/Pickup 诊断模板

| 现象 | 初步判断 | 推荐动作 | 验证数据 |
| --- | --- | --- | --- |
| Pace 落后，Pickup 也慢 | 需求不足或价格/流量承接失败 | 查竞对价格、查流量下滑点、评估促销或渠道补量 | 商圈流量、竞对价、曝光-浏览-订单转化 |
| Pace 落后，但 Pickup 加速 | 仍有追赶机会 | 先观察，避免过早大幅降价 | 未来 24-72 小时订单增速 |
| Pace 正常，但未来 14 天出现空窗 | 局部日期有库存风险 | 对空窗日期做定向促销、房型排序或会员触达 | 日期级订单、房型库存、价格带 |
| Pickup 突然暴增 | 可能出现展会、演唱会或突发需求 | 锁定高价库存，逐步提价并观察成交 | 城市事件、竞对价格、成交间隔 |
| 高 OCC 但低 ADR | 量够但收益质量不足 | 关停低价渠道或提升低价房型门槛 | 渠道结构、房型结构、取消率 |

### 2.3 动态价格 SOP

每日晨间收益巡查建议拆成 5 步：

1. 查看今日及未来 14 天 Pace/Pickup。
2. 检查 AI 策略优先级：流量问题、转化问题、价格问题、库存问题。
3. 查看竞对实时价格和商圈排名，确认我方价格位置。
4. 如需调价，按价格等级、小幅多次、日期/房型/渠道分层执行。
5. 记录调价原因、数据证据、执行人和复盘时间。

执行守卫：

| 守卫 | 规则 |
| --- | --- |
| 不拍脑袋 | 没有 Pace、Pickup、竞对或库存证据时，不给大幅调价建议 |
| 不裸奔 | 不用公开低价破坏品牌，应优先做会员价、套餐、连住、早订等分层动作 |
| 不恋战 | 价格战中先止损、保现金流，再看是否继续对抗 |
| 不自动执行 | AI 只能给建议和证据，调价必须经过人工确认和底价校验 |

### 2.4 节假日盯盘框架

| 时间点 | 经营动作 | 数据要求 |
| --- | --- | --- |
| D-30 | 建价格日历、设目标 OCC/ADR、锁定竞对集合 | 历史节假日数据、城市事件、竞对价 |
| D-14 | 观察 Pace 是否符合预期，必要时启动早鸟或预售 | 预订进度、Pickup、渠道结构 |
| D-7 | 进入高频盯盘，按日调整价格等级和库存 | 早/午/晚三次巡查记录 |
| D-3 | 识别空房危机或满房预警 | OCC、可售房、竞对可售、取消率 |
| D-0 至 D+1 | 动态关停低价渠道、处理临期库存、记录复盘样本 | 成交速度、价格等级、订单质量 |
| 节后 | 复盘目标差异、成功动作、失败动作和下一次阈值 | RevPAR、ADR、OCC、渠道成本、客诉 |

资料中提到的具体阈值，例如 D-3、OCC、Pace 和涨价幅度，只能作为经验参数候选。宿析OS应先把它们放入“待验证策略参数”，用历史节假日样本回测后再启用。

### 2.5 竞对价格战应对

当竞对用低价抢量时，不建议直接一刀切跟降。更稳的流程：

1. 先确认竞对低价覆盖哪些日期、房型、渠道和库存量。
2. 判断我方损失来自曝光下降、转化下降、价格劣势还是库存展示问题。
3. 用会员价、老客权益、连住、套餐、房型升级等方式做分层响应。
4. 对低转化渠道止损，对高转化渠道精准补量。
5. 72 小时内连续复盘出租率、ADR、订单质量和取消率。

## 三、映射到宿析OS

### 3.1 指标字段建议

| 知识指标 | 计算口径 | 数据来源优先级 | 数据状态 |
| --- | --- | --- | --- |
| `occupancy_rate` | 售出间夜 / 可售间夜 | PMS/日报房态 > 人工导入 > OTA 部分订单 | 分母缺失则不可计算 |
| `market_share` | 本店间夜 / 商圈同类间夜 | 平台商圈排名/竞对摘要 > 第三方市场数据 > 人工竞对池 | 竞对分母缺失则不可计算 |
| `pace_rate` | 当前累计预订 / 目标基线 | 本店订单 + 历史同期/近期均值 | 基线缺失则只能显示累计值 |
| `pickup_rate` | 近期新增预订或间夜 | 订单增量、渠道订单、OTA 订单 | 需要时间窗口 |
| `d_pace` | 当日分小时预订速度 | 当日订单流、OTA 实时订单 | 需要小时级时间戳 |
| `adr` | 房费收入 / 间夜 | 财务/PMS > OTA 平台收入 | 必须标注全店或平台口径 |
| `revpar` | 房费收入 / 可售房间数 | 财务/PMS + 房量 | OTA 收入不能冒充全店 RevPAR |
| `flow_conversion` | 曝光 -> 浏览 -> 订单/支付/入住 | 携程/美团后台、`raw_data`、平台摘要 | 平台口径分开计算 |

### 3.2 AI 决策输入

AI 给收益建议时，至少需要以下证据：

| 证据 | 示例 |
| --- | --- |
| 门店范围 | 当前门店、集团门店、竞对池 |
| 时间范围 | 今日、未来 7 天、未来 14 天、节假日周期 |
| 数据来源 | 携程、美团、PMS、日报、人工导入、用户资料 |
| 指标口径 | OTA 平台口径、全店口径、商圈口径 |
| 置信状态 | 已核验、用户资料未核验、数据缺口、不可计算 |
| 建议动作 | 调价、促销、关低价渠道、补库存、会员触达、观察 |
| 执行约束 | 底价、品牌价、房型库存、渠道规则、人工确认 |

## 四、可执行检查清单

### 4.1 每日收益巡查

- 今日和未来 14 天 Pace 是否低于基线。
- 最近 3 天 Pickup 是加速、放缓还是异常突增。
- 是否存在日期级空窗、房型级空窗或渠道级空窗。
- 本店价格是否明显偏离竞对价格带。
- 曝光、浏览、订单、取消、入住哪个环节异常。
- 是否需要调价、促销、关低价渠道、补库存或继续观察。
- 每个动作是否记录了数据证据和复盘时间。

### 4.2 节假日前检查

- 是否有节假日价格日历。
- 是否拆解 OCC、ADR、RevPAR 目标。
- 是否建立竞对池和事件日历。
- 是否设置底价、封顶价和低价渠道关闭条件。
- 是否安排 D-14、D-7、D-3、D-0 巡查节奏。
- 是否有节后复盘模板。

### 4.3 集团多店检查

- 各门店是否使用同一指标口径。
- 是否区分不同城市、商圈和竞对池。
- 是否能看到每店 Pace/Pickup、商圈位置和价格带。
- 是否有每周收益例会的决策记录。
- 是否沉淀可复用 SOP，而不是只在群里口头沟通。

## 五、待验证项

| 待验证项 | 原因 | 建议验证方式 |
| --- | --- | --- |
| 行业新增酒店、OCC 下滑、佣金、人力和成本涨幅 | PDF 为用户资料，未核验来源 | 联网行业报告、平台后台合同、门店财务数据 |
| 市占率分母 | 商圈总客源/间夜通常不是直接可得 | 先用平台竞对摘要或人工竞对池标记为估算 |
| Pace/Pickup 阈值 | 不同城市、商圈、门店体量不同 | 用本店历史样本回测，按门店配置阈值 |
| 节假日调价阈值 | 节假日类型差异大 | 按节假日、城市事件和房型做差异化参数 |
| AI 调价建议有效性 | 需要闭环样本证明 | 保存建议、执行、结果、复盘，形成策略评估集 |
| 自动调价合规性 | 涉及底价、品牌价、平台规则和人工审批 | 先做人审建议，不做默认自动执行 |

## 六、结论

这份资料适合沉淀为“收益经营方法论”，核心不是某一个工具，而是把酒店经营从经验判断转为可追溯的数据闭环：

```text
预订进度 -> 竞对位置 -> 价格/渠道动作 -> 执行记录 -> 结果复盘 -> SOP 固化
```

宿析OS吸收时要坚持三点：

1. OTA 数据只代表平台/渠道口径，不能替代全店出租率和全店 RevPAR。
2. 阈值、案例结果和行业数据先标记为 `user_provided_unverified`。
3. AI 只输出带证据和约束的建议，执行必须经过人工确认、底价校验和复盘记录。
