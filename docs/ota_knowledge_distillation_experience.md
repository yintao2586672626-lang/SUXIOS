# OTA知识沉淀经验总结

## 适用场景

当资料来自图片、旧版培训材料、平台后台截图、Excel 导出、人工经验或联网资料时，宿析OS应先把信息沉淀为“可追溯、可计算、可诊断、可执行”的知识，再决定是否进入业务字段、报表或自动化流程。

本经验来自 2026-05-26 对 `E:\杂项\OTA运营思维导图2.0版` 的处理，以及后续将 OTA 指标联网校验并引入宿析OS知识库的过程。

## 核心经验

| 经验 | 说明 | 宿析OS落地 |
| --- | --- | --- |
| 先分清资料类型 | 图片思维导图、平台规则、行业标准、项目既有口径不能混为一类 | 图片先 OCR，旧资料标注时效，行业指标找公开公式，项目口径保留兼容 |
| 先沉淀知识，不急着改表 | 大量运营知识适合进入知识库，不一定适合立即结构化成业务表 | 优先写入 `knowledge_units`、`knowledge_chunks`、`knowledge_base` |
| 指标必须有口径 | 同一个指标在全店、OTA渠道、平台、房型、广告活动中的含义不同 | 每条指标必须标注 `metric_scope` 和 `calculation_basis` |
| 标准指标用权威公式 | ADR、入住率、RevPAR、CTR、转化率等应使用行业或平台公开口径 | 公式写入知识库，计算时分母缺失返回不可计算 |
| 平台私有分值不反推 | 携程服务质量分、美团 HOS、飞猪 MCI 等可能有私有权重 | 保存后台值、含义、影响因素和行动建议，不写死公式 |
| 旧资料与新资料要合并别名 | 旧图中的 `MIC` 与公开常见 `MCI` 可能指向同类飞猪服务指标 | 系统统一字段 `fliggy_mci`，保留 `MIC` 作为历史别名 |
| 运营建议必须能落到动作 | “提升流量”太泛，必须拆成排名、标签、首图、库存、价格、平台分等动作 | 知识块中保留“现象-判断-动作”诊断模板 |
| 验证要覆盖幂等和初始化 | 知识 seed 必须可重复执行，且 `init_full.sql` 不引用缺失文件 | 临时库执行两次，验证 unit/chunk/category/base 不重复 |

## 推荐流程

1. 资料盘点
   - 列出文件来源、文件类型、数量和日期。
   - 对旧资料标注“执行前需复核当前平台后台”。
   - 不根据文件名或模糊截图直接编结论。

2. 原文抽取
   - 图片资料先 OCR，并保存原始抽取结果。
   - 对 OCR 错字只做可验证修正，不凭空补内容。
   - 抽取结果先按主题归类，再总结。

3. 业务归类
   - OTA 运营知识优先按平台、流量、转化、内容、点评、房态、库存、评分、数据诊断分类。
   - 指标知识优先按经营收益、OTA流量、交易转化、价格竞争、库存房态、口碑服务、用户行为、投放活动分类。

4. 联网校验
   - 行业通用指标优先找酒店行业、PMS、广告平台、OTA/分销平台公开文档。
   - 对平台私有分值，只确认公开含义和影响范围，不推导未公开权重。
   - 把来源链接写入知识块，方便后续复核。

5. 项目融合
   - 不覆盖既有项目口径，先比较差异。
   - 行业标准公式优先；项目字段边界优先；旧资料作为运营经验保留。
   - 对别名、历史叫法、平台差异做兼容说明。

6. 知识落库
   - 人读版本写入 `docs/`。
   - 系统检索版本写入 migration seed。
   - 同时写 `knowledge_units`、`knowledge_chunks` 和 `knowledge_base`。
   - chunk 要结构化，便于 Agent 检索；knowledge_base 要简短，便于员工阅读。

7. 验证闭环
   - `init_full.sql` 必须 SOURCE 新 seed。
   - 运行 `DatabaseBuildScriptTest`。
   - 用临时 MySQL 库执行 seed 两次，验证幂等。
   - 检查新增文件无行尾空格，确认没有敏感信息进入 Git。

## 知识库结构建议

| 层级 | 内容 | 适合字段 |
| --- | --- | --- |
| `knowledge_units` | 一个完整知识主题 | 名称、来源、状态、描述、标签 |
| `knowledge_chunks` | 可检索的结构化片段 | 使用边界、指标清单、诊断模板、落地规则、来源 |
| `knowledge_base` | 员工可读摘要 | Markdown 摘要、关键词、标签 |
| `docs/` | 长文档、方法论、完整解释 | 资料边界、详细表格、验证说明 |

## 质量守卫

- 不把 OTA 渠道数据冒充全店经营数据。
- 不把旧平台规则写成当前官方规则。
- 不把缺分母指标写成 0。
- 不在没有广告成本时输出 ROI。
- 不反向工程平台私有分数。
- 不把 Cookie、token、账号、手机号、订单明细、点评明细等敏感信息写入普通文档或 seed。
- 不为了“完整”新增业务表；只有页面、报表、预警或 AI 分析明确需要结构化读取时才新增字段。

## 可复用清单

每次做知识沉淀前检查：

- 资料来源是否明确。
- 资料是否过期或可能变化。
- 是否有原始抽取证据。
- 是否区分行业标准、平台规则、项目口径和运营经验。
- 是否标注公式、分母、粒度、适用范围和不可计算条件。
- 是否能映射到现有字段或 `raw_data`。
- 是否写入知识库三层结构。
- 是否完成幂等验证。
