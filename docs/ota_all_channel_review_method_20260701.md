## 二、可沉淀的核心方法

### 2.1 先建“点评证据链”，再给运营建议

每条点评相关判断必须先确认：

| 证据项 | 要求 |
| --- | --- |
| `platform` | 携程、美团、去哪儿、同程、Trip.com 等 |
| `hotel_id` | 宿析OS内部门店 ID |
| `review_source_status` | `user_provided_unverified`、`platform_backend_verified`、`official_rule_verified` |
| `review_time` | 点评发布时间，缺失时标为 `unknown` |
| `stay_time` | 入住/离店时间；仅在合法授权后台可见时保存 |
| `score` / `sub_scores` | 原始平台分值和子项分，保留原始分制 |
| `content_tags` | 卫生、服务、设施、噪音、价格、订单、竞对、不可核验等 |
| `score_count_mode` | 计分、仅展示、第三方同步、追评、未知 |
| `display_mode` | 正常展示、折叠、置底、隐藏、未知；必须有平台证据 |
| `operator_action` | 回复、整改、申诉、客服沟通、客诉补救、观察 |
| `action_evidence` | 截图、后台记录、客服工单、整改照片、回复文本 |
| `next_check_at` | 复核时间 |

缺少关键证据时，AI 只能输出“证据不足”和补证清单，不能输出确定性删评、改分、排名或流量结论。

### 2.2 点评问题分类

| 类型 | 判断重点 | 推荐处理 |
| --- | --- | --- |
| 服务体验问题 | 态度、响应速度、投诉处理、入住沟通 | 门店复盘、员工培训、客户补救、回复模板 |
| 卫生问题 | 客房、布草、卫浴、异味、虫害 | 整改任务、复检照片、班组责任、复盘周期 |
| 设施问题 | 空调、热水、隔音、网络、床品 | 工程维修、房态冻结、复检后恢复 |
| 订单/支付问题 | 价格、房型、取消、发票、押金 | 前台 SOP、订单备注、退款/补偿记录 |
| 平台展示/规则问题 | 计分、展示、折叠、申诉、同步 | 以平台后台规则和客服工单为准 |
| 疑似违规/竞对问题 | 明显不实、辱骂、广告、竞对攻击 | 只走平台合规申诉，保留证据，不做反制骚扰 |
| 不可核验问题 | 无订单线索、内容泛化、证据不足 | 标为 `unverified`，不进入确定性结论 |

### 2.3 点评分到流量的使用方式

PDF 中提到携程点评分层级与流量的关系：

| 点评分 | PDF 描述的流量信号 | 使用状态 |
| --- | --- | --- |
| 4.0 | 挂牌流量 | 用户资料口径，待后台核验 |
| 4.5 | 榜单流量 | 用户资料口径，待后台核验 |
| 4.7 | PSI 流量、筛选流量 | 用户资料口径，待后台核验 |
| 4.9 | 好评优先、排序流量 | 用户资料口径，待后台核验 |

系统只能把它作为“待核验影响假设”，不能写死为平台排序公式。收益分析中应继续结合曝光、点击、转化、价格、库存、竞对和订单数据。

### 2.4 差评处理闭环

```text
差评出现
-> 核实平台、订单、点评时间、内容标签、计分/展示状态
-> 判定真实问题 / 违规问题 / 不可核验问题
-> 真实问题：回复 + 补救 + 整改任务 + 复检
-> 违规问题：证据清单 + 平台申诉 + 工单状态记录
-> 不可核验：标注证据不足，不做确定性处理
-> 复核分数、展示状态、转化影响
```

差评处理的目标不是“消灭差评”，而是减少真实服务问题、修复客诉、提升真实口碑质量，并把平台申诉作为合规通道。

### 2.5 好评增长原则

| 可做 | 不可做 |
| --- | --- |
| 提升真实入住体验 | 伪造入住、伪造订单、刷好评 |
| 在离店后合规提醒客人评价 | 诱导、强迫、利益交换式好评 |
| 引导客人描述真实体验、上传真实图文 | 复制粘贴模板、AI 生成虚假内容 |
| 关注近期真实好评数量和质量 | 用低权重、异常订单堆数量 |
| 复盘差评原因，减少问题复发 | 用覆盖、隐藏、规避机制替代整改 |

PDF 中关于“不是所有 5 分都能涨分”“真实账号、真实佣金、真实入住、真实点评”的部分，可以沉淀为好评质量原则；任何非真实、诱导或规避平台风控的做法不能进入系统方法。

### 3.1 风险知识沉淀：只学习识别，不学习执行

这些知识可以进入宿析OS的“风险识别”和“AI输出守卫”，用于判断请求是否越界、是否需要转为合规处理。不得沉淀成执行步骤、脚本、自动化、员工 SOP 或默认运营建议。

| 风险知识 | 系统需要学会识别什么 | 禁止输出 | 合规替代 |
| --- | --- | --- | --- |
| 身份反查 | 试图通过头像、昵称、用户名、订单时间、IM成员、加密字符串等线索推断点评人真实身份 | 反查路径、匹配技巧、枚举逻辑、字段选择器、自动化脚本 | 只记录平台合法可见的脱敏证据状态；问题处理回到点评内容和服务整改 |
| 手机号获取 | 试图通过后台按钮、页面漏洞、客服话术、关联订单等方式获取或补全客人手机号 | 获取号码步骤、绕过展示限制、批量提取方法 | 使用平台允许的站内沟通、订单已授权联系方式和客服工单；保存最小必要字段 |
| 匿名用户匹配 | 试图把匿名点评与住客、同行人、IM会话、会员等级、入住时间做交叉定位 | 匿名还原方法、候选人筛选、相似度/枚举策略 | 标记为 `anonymous_unverified`；只做内容分类、客诉补救和通用整改 |
| 规避点评 | 试图通过取消、删除、补单、改订单状态、切换支付方式等方式避免客人评价或影响计分 | 规避平台评价机制的操作流程 | 正常处理客诉；必要时按平台规则取消、退款、补偿，并保留原因和证据 |
| 覆盖差评 | 试图通过设计订单、低权重订单、异常好评、诱导评价等方式稀释或压制差评 | 刷评、诱导、覆盖、权重操控方案 | 真实好评增长、服务整改复盘、合规评价提醒、差评回复和申诉材料 |

### 3.2 AI 拦截规则

当用户或员工提出以下意图时，AI 应转为合规解释和替代方案：

| 触发意图 | AI 应答 |
| --- | --- |
| “帮我查这个差评是谁” | 不能协助身份反查；可协助按点评内容、订单范围和服务问题做证据清单 |
| “怎么拿到匿名用户手机号” | 不能协助获取或补全手机号；可协助使用平台站内沟通和客服工单 |
| “怎么匹配匿名点评和订单” | 不能输出匿名匹配方法；可输出 `anonymous_unverified` 标注和整改复核流程 |
| “怎么让客人不能点评” | 不能协助规避点评；可协助处理客诉、退款、取消和平台规则复核 |
| “怎么用好评压下差评” | 不能协助覆盖差评；可协助制定真实好评增长和差评原因整改计划 |

推荐应答结构：

```text
这个请求涉及隐私反查/平台规则规避，不能作为宿析OS执行方法。
可以改为：
1. 核实点评内容与平台证据状态。
2. 标注是否计分、是否展示、是否可申诉。
3. 对真实问题生成整改和回复。
4. 对疑似违规点评整理合规申诉材料。
5. 在复核时间点观察评分、展示和转化变化。
```

### 3.4 外部常见做法的抽象模式

本节用于理解行业里可能存在的做法，帮助识别风险来源；不是操作指南。

| 抽象模式 | 别人通常想达到的目的 | 常见信号 | 宿析OS应如何吸收 |
| --- | --- | --- | --- |
| 跨页面身份拼接 | 把点评页的弱身份线索与订单、IM、会员、头像、昵称等页面线索拼起来，推断点评人 | 需要跨页面、跨时间、跨字段比对；结果通常只有概率，没有平台确认 | 学习为 `identity_linking_risk`，只允许做风险提示，不把推断身份写入经营事实 |
| 脱敏信息补全 | 从被隐藏的用户名、手机号、昵称尾号等片段推测完整身份 | 依赖枚举、候选集、相似度或人工逐个排查 | 学习为 `masked_data_reconstruction_risk`，禁止输出补全方法 |
| 利用客服或后台入口补证 | 通过客服、工单或后台可见字段尝试获取更多订单或客人信息 | 以“处理差评”为名扩大个人信息访问范围 | 学习为 `over_collection_risk`，只保留合规工单号、处理状态和脱敏证据 |
| 订单状态干预 | 通过取消、删除、补单、支付方式变化等影响点评资格或计分状态 | 动作目标不是解决客诉，而是改变点评入口或评分后果 | 学习为 `review_lifecycle_evasion_risk`，转为客诉处理和平台规则复核 |
| 好评稀释/覆盖 | 用新增好评数量或特定订单类型降低差评展示和转化影响 | 关注“压下去”“盖过去”“快速涨分”而不是服务问题 | 学习为 `rating_manipulation_risk`，转为真实好评增长和整改闭环 |
| 申诉包装 | 把不一定违规的差评包装成竞对、恶意、虚假，尝试让平台折叠或隐藏 | 证据不足但结论先行，要求保证处理成功 | 学习为 `appeal_overclaim_risk`，要求先列证据，不承诺结果 |

对这些模式的正确学习方式是：

```text
知道它存在
-> 识别它的业务动机
-> 标记隐私/平台/合规风险
-> 禁止生成可执行步骤
-> 给出合规替代路径
-> 把结果回到点评事实、服务整改、申诉证据和收益影响观察
```

| IM 会话身份推断 | 利用系统欢迎语、回复窗口、会话留存、房型/头像/地址对比推断住客 | 诱导进对话框、批量保存会话、通过 IM 还原身份 | 使用平台站内客服处理客诉，不做身份反推 |
| 加密标识枚举 | 用订单号、姓名片段、隐藏用户名或哈希值尝试还原真实账号 | MD5/哈希猜解、枚举字典、外部破解工具、命中判断 | 将该类请求标记为 `masked_data_reconstruction_risk` |
| 点评人筛选 | 在差评出现后反向定位具体住客或真实用户名 | 差评人定位、匿名用户匹配、真实用户名输出 | 输出 `anonymous_unverified`，转为证据清单、回复、申诉和整改 |

AI 守卫补充：

1. 任何“登记所有订单客人手机号/微信”“找到匿名点评是谁”“根据头像/IM/MD5查客人”的请求，必须拦截为隐私反查风险。
2. 允许输出的是合规替代：平台站内沟通、客服工单、已授权联系方式、客诉补救、整改复核、脱敏证据状态。
3. 禁止输出字段路径、外部工具、哈希计算、候选枚举、头像比对、批量保存会话或账号还原步骤。
4. 如果业务需要联系客人，必须以平台授权、客人同意、最小必要、可审计为前提。

可学习的风险模式：

| 风险模式 | 识别信号 | 系统处理 |
| --- | --- | --- |
| 差评后反向找人 | 目标从“处理服务问题”变成“找到差评是谁” | 拦截为 `reviewer_identification_risk`，只允许做点评事实和整改清单 |

| 客户端/安装包分析 | 使用客户端包、页面元素、网络字段或内部标识辅助定位用户 | 标记 `platform_reverse_engineering_risk`，禁止输出分析路径 |

合规替代流程：

```text
差评出现
-> 记录平台、时间、分值、问题类型和证据状态
-> 判断是否存在服务/卫生/设施/订单/价格问题
-> 能核验则做客诉补救、回复、整改、申诉材料
-> 不能核验则标记 anonymous_unverified
-> 复盘曝光、浏览、转化、订单影响
-> 只用聚合趋势进入收益和投决分析
```

系统功能只能做：

1. 点评问题分类和证据完整度检查。
2. 合规回复、整改任务、申诉材料辅助。
3. 平台授权范围内的沟通状态记录。
4. 脱敏后的口碑趋势和收益影响观察。
5. AI 输出守卫：拦截手机号获取、微信登记、匿名身份匹配、客户端逆向、哈希/标识猜解和跨页面用户拼接。

## 五、AI 输出守卫

点评相关 AI 建议必须遵守：

1. 先说明平台和证据状态。
2. 先区分 OTA 渠道口径和全酒店经营口径。
3. 缺少订单、时间、分值、展示状态、平台规则时，不能给确定性结论。
4. 不承诺删评、隐藏、恢复分数、提升排名或获得某类流量。
5. 不生成身份反查、电话获取、匿名用户匹配、诱导好评、规避差评的操作步骤。
6. 能输出的动作限于：事实核验、客诉补救、服务整改、合规回复、平台申诉材料、复核计划、收益影响观察。

## 六、结论

这份 PDF 对宿析OS的价值不是“照做点评技巧”，而是帮助建立一套可验证、可追溯、合规的点评运营方法：

```text
点评事实采集
-> 平台规则和证据状态标注
-> 服务问题分类
-> 收益/转化影响观察
-> AI 生成合规运营动作
-> 门店整改和复核
-> 投资决策只引用聚合趋势，不引用单条点评推断
```

可沉淀的是点评影响认知、证据链、问题分类、申诉边界、真实好评增长和服务整改闭环。

不可沉淀的是隐私反查、规避平台点评机制、刷评、覆盖差评、自动删评或任何会掩盖真实服务问题的做法。
