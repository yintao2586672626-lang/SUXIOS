# 宿析OS时间认知体系

## 一句话原则

> 过去有据，如今可察，未来可观。

这句话是宿析OS的数据与决策契约，不是宗教视觉主题：过去必须能追溯证据，如今必须反映最新有效状态，未来必须是带来源、置信度和版本的可能性判断。

## 产品目的

宿析OS需要让用户每天看到真实变化，并在未来回看“当时系统知道什么、判断了什么、后来实际发生了什么”。系统不依靠焦虑文案制造使用频率，而依靠连续、可复盘的数据价值形成使用习惯。

```text
过去事实 -> 如今快照 -> 未来预测 -> 到期实际 -> 预测复盘 -> 新的历史证据
```

## 四类时间对象

| 对象 | 业务含义 | 默认来源 | 保存规则 | 页面规则 |
|---|---|---|---|---|
| 过去 `past` | 已发生且可验证的最终事实 | `daily_reports`；或 `online_daily_data` 中 `historical_daily + is_final=1` 的 OTA 事实 | 同一业务键、同一日期保留最终事实；修正必须留下来源与更新时间 | 实线展示，可查看来源、日期和质量状态 |
| 如今 `present` | 截至当前采集时点的最新有效状态 | `online_daily_data` 中 `realtime_snapshot` | 按采集任务或分钟形成快照；页面按业务键取最新有效版本 | 以“截至 HH:mm”展示，不用昨日数据冒充今日 |
| 未来 `future` | 基于过去事实和当前条件形成的趋势判断 | 历史事实的确定性计算；可辅以已验证 OTA 未来需求信号 | 每次正式生成形成新版本，记录 `as_of_time`、`target_date`、方法和输入证据 | 虚线或区间展示，必须标注置信度与渠道范围 |
| 复盘 `review` | 将当时预测与后来最终事实对齐 | 已保存预测版本 + 到期后的最终事实 | 不覆盖旧预测；实际缺失时保持待复盘 | 显示方向命中、区间覆盖和偏差，不虚构成功 |

## 时间身份

任何进入统一时间轴的数据至少保留：

- `tenant_id`、`system_hotel_id`：租户与系统门店边界。
- `metric_scope`：默认 `ota_channel`；没有全量经营数据时不得写成全酒店事实。
- `platform`、`metric_key`：渠道和指标身份。
- `business_date` 或 `target_date`：指标归属日期或预测目标日期。
- `as_of_time`：系统在什么时间知道这些信息。
- `source_trace_id` 或 `source_refs`：证据来源。
- `data_quality_status`：`available/partial/stale/unverified/collection_failed` 等真实状态。
- `model_version`、`confidence_score`：预测方法与可信程度。

## 事实优先级与边界

1. 全酒店事实只使用可验证的 PMS/日报/全渠道口径。
2. OTA数据默认只描述对应渠道，不外推为全酒店出租率、ADR、RevPAR或收入。
3. 历史趋势只读取最终事实；实时快照和未来预测不得参与历史累计。
4. 今日状态只读取目标日期的最新有效快照；缺失时显示缺口，不回填旧日数据。
5. 缺失分母、采集失败和未返回字段保持 `null` 或缺口状态，不转换成 `0`。
6. AI只解释已计算结果，不生成指标、不执行调价、不把建议写成事实。

## 粗粒度预测契约

首版预测面向 OTA 收益、订单、间夜等可验证日级指标，不生成具体执行价格。

- 历史窗口：默认近28天，最少7个有效日。
- 比较结构：最近7天、前7天、同星期特征和波动程度。
- 输出：方向、预测区间、置信度、样本数、数据截止时间和方法版本。
- 置信度：由样本覆盖、新鲜度、波动程度和数据质量共同决定。
- 样本不足：返回 `insufficient`，不使用默认值补齐。
- 表达边界：使用“若现有条件延续”“可能”“区间”等语言，不承诺结果。

## 三段式解释

所有面向用户的时间解读统一为：

1. **过去**：历史基线和关键变化是什么。
2. **如今**：截至当前时点，指标处于什么状态。
3. **未来**：若当前条件延续，未来可能如何变化。

只有真实快照之间存在新增异常、趋势转向或数据盲区时，才提示“今日值得查看”。不得使用虚构倒计时、夸大损失或无数据支撑的焦虑文案。

## 为什么不采用其他方案

### 不做三个割裂页面

用户真正需要的是同一个指标如何从过去演变到如今，再进入未来。拆成三个页面会迫使用户自行拼接口径，复盘也无法自然发生。宿析OS使用一个时间轴、三个时间视角。

### 不把所有数据继续放进同一趋势累计

事实、实时快照和预测的时间含义不同。混合汇总会重复计算今日快照，并把未来目标日期误当成历史事实，数据越多反而越不可信。

### 不直接让LLM预测数字

预测基线由确定性代码计算，方便回放、测试和比较版本。AI只把结果解释成人能理解的语言，避免不可复现的数字变化。

### 不做精确自动定价

当前数据和执行证据不足以支撑自动调价。价格只作为观察信号；任何价格动作继续走人工确认与执行证据闭环。

### 不使用宗教图形和术语包装

“过去、如今、未来”作为底层认知结构存在，界面保持高端酒店经营系统的专业与包容，不使用莲花、佛像、经文或玄学式表达。

## 首个闭环验收

1. 同一门店同一指标在同一天多次查询，可以区分采集时点并完成数据库回读。
2. 历史曲线每个业务键每天只计一次最终事实。
3. 今日状态显示最新有效快照和明确的“截至时间”。
4. 未来预测具有生成时间、目标日期、版本、区间、置信度和来源范围。
5. 目标日期到期后，可以从历史日期打开“回看当时”，对照当时预测与最终事实。
6. 缺失、过期、未验证和采集失败均真实展示。
7. 至少一家授权门店完成查询、保存、回读、页面展示和复盘验证。
