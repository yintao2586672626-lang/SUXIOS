# Release Blocker Close Plan

Updated: 2026-07-06

Scope: `@github`, `@openai-developers`, `@codex-security`, `@figma`, `@canva`

Source: `docs/release_readiness_status.json` `blockers`

Current status: `production-env-missing`, `llm-connectivity-attestation-missing`, and `codex-security-scan-missing` are closed by external/repo evidence and their isolated review commands. `design-handoff-missing`, `ota-credential-rotation-attestation-missing`, and `local-git-state-open` remain open. The GitHub handoff blocker must stay open until real design and OTA evidence pass their isolated gates, an actual open final PR exists, `npm run review:release-pr-candidates` selects it, staged PR scope passes `npm run review:release-staged-scope`, and release-readiness consumes passing staged-scope and external-state results.

Command matrix: `docs/release_verification_command_matrix.md`

## Close Rules

- Rerun `npm run review:release-readiness` after closing each blocker.
- For production env review, run `npm run review:release-env` first; it uses the same env rules as `npm run review:release-readiness` without requiring the unrelated release artifacts to be present.
- For production LLM review, run `npm run review:release-llm` first; it uses the same attestation rules as `npm run review:release-readiness` without requiring the unrelated release artifacts to be present.
- For Figma / Canva handoff review, run `npm run review:release-design` first; it uses the same manifest rules as `npm run review:release-readiness` without requiring the unrelated release artifacts to be present.
- For OTA credential and backup cleanup review, run `npm run review:release-ota-credentials` first; it uses the same backup scan and rotation-attestation rules as `npm run review:release-readiness` without requiring the unrelated release artifacts to be present.
- For Codex Security scan review, run `npm run review:release-security-scan` first; it uses the same scan artifact rules as `npm run review:release-readiness` without requiring the unrelated release artifacts to be present.
- `npm run review:release-evidence` writes `release-evidence-result.json` to `RELEASE_EVIDENCE_RESULT_FILE` or to `RELEASE_EVIDENCE_DIR` / `../release-evidence-temp` by default, outside the repository; this preflight result does not close release by itself.
- `npm run review:release-evidence-drafts` writes `release-evidence-draft-review-result.json` to `RELEASE_EVIDENCE_DRAFT_REVIEW_RESULT_FILE` or to `RELEASE_EVIDENCE_DIR` / `../release-evidence-temp` by default, outside the repository; it only decides whether external drafts are eligible for guarded promotion and must not be treated as final release evidence.
- `npm run promote:release-evidence-drafts` writes `release-evidence-promotion-result.json` or `RELEASE_EVIDENCE_PROMOTION_RESULT_FILE` outside the repository on pass or fail; the result is an audit trail only and does not close release readiness unless the final manifest/attestation files are created and their isolated acceptance commands pass.
- `npm run release:create-design-manifest` can either validate and copy a complete external `DESIGN_HANDOFF_MANIFEST_INPUT_FILE`, or compile from release-closing Figma / Canva connector evidence plus `DESIGN_HANDOFF_OWNER` and `CANVA_BRAND_KIT_URL`. It writes `release-design-manifest-create-result.json` or `DESIGN_HANDOFF_MANIFEST_CREATE_RESULT_FILE` outside the repository on pass or fail, including input/candidate/output SHA-256 fingerprints; the result is an audit trail only and does not close release readiness unless the final `design_handoff_manifest.json` exists and `npm run review:release-design` passes.
- `npm run release:create-ota-attestation` validates and copies a complete external `OTA_CREDENTIAL_ROTATION_INPUT_FILE`; the input must be outside the repository, not a draft, and not the same path as the final output. It writes `release-ota-attestation-create-result.json` or `OTA_CREDENTIAL_ROTATION_CREATE_RESULT_FILE` outside the repository on pass or fail, including input/output SHA-256 fingerprints; the result is an audit trail only and does not close release readiness unless the final `ota_credential_rotation_attestation.json` exists and `npm run review:release-ota-credentials` passes.
- `npm run report:release-evidence-gap-pack` writes a machine-readable `operator_intake_packet` and `local_worktree_close_plan` outside the repository. Use the intake packet to collect real external design, OTA credential-rotation, and final PR/local-state inputs; use the worktree plan to separate release docs/scripts from frontend, revenue-AI, runtime/local, and other changes before selecting the final PR. Runtime, storage, reports, and local evidence must not be committed by default.
- `npm run verify:release-evidence-gap-pack` verifies only that the generated gap pack is outside the repository, structurally complete, and explicitly non-closing. It must preserve the current GitHub connector open-PR count as evidence, but the count is not fixed to zero because a future final PR must be selectable before external-state can close.
- `npm run export:release-operator-intake` writes `release-operator-intake-packet.json` and `release-operator-intake-packet.md` from the latest gap pack for handoff to the accountable operator. This packet is a checklist only; it must not be treated as final design handoff, OTA credential rotation, PR, local-state, or release-readiness evidence.
- `npm run verify:release-operator-intake` verifies only that the exported operator packet is outside the repository, structurally complete, and explicitly non-closing. It does not replace final design, OTA, PR, local-state, or release-readiness evidence.
- `npm run refresh:release-current-evidence` refreshes `worktree-quarantine-current/manifest.json` and the release-staging TSVs before writing the gap pack, so the local-state blocker points at current dirty-worktree evidence; that quarantine output is preservation evidence only and still requires a clean worktree or explicitly reviewed final PR.
- `npm run review:release-readiness` writes `release-readiness-result.json` to `RELEASE_READINESS_RESULT_FILE` or to `RELEASE_EVIDENCE_DIR` / `../release-evidence-temp` by default, outside the repository; archive that generated JSON outside secret-bearing locations. A pre-ready result must be treated as `mode=pre_ready`, not final release-ready evidence.
- For GitHub/local-state review, run `npm run review:release-pr-candidates` to discover open viable release PRs; if exactly one viable PR exists it is selected automatically, and if multiple viable PRs exist set `RELEASE_PR_NUMBER` to the intended final PR and rerun `npm run review:release-pr-candidates` so the result records both `configured_release_pr_number` and `selected_release_pr_number`. The command writes `release-pr-candidates-result.json` to `RELEASE_PR_CANDIDATES_RESULT_FILE` or to `RELEASE_EVIDENCE_DIR` / `../release-evidence-temp` by default, outside the repository, and records the current `gh pr list` source, `gh_pr_list_checked_at`, `gh_pr_list_open_pr_count`, attempts, and `docs/release_github_handoff_evidence.json` / `RELEASE_GITHUB_HANDOFF_EVIDENCE_FILE` connector diagnostics with `open_pr_count`, `checked_at`, and stale/non-closing flags. The current `gh pr list` result is the PR candidate source of truth; connector evidence is diagnostic-only and does not close final PR state. Before final PR staging, run `npm run review:release-staged-scope`; it writes `release-staged-scope-result.json` to `RELEASE_STAGED_SCOPE_RESULT_FILE` or the same controlled evidence directory and rejects runtime/local staged files. Then run `npm run review:release-external-state`; it writes `release-external-state-result.json` to `RELEASE_EXTERNAL_STATE_RESULT_FILE` or to `RELEASE_EVIDENCE_DIR` / `../release-evidence-temp` by default, outside the repository, and now requires the captured local `git rev-parse HEAD` to match the selected final PR head. If Node child_process is blocked, use `npm run review:release-external-state:local` so the PowerShell wrapper writes both collected evidence and the result JSON under the controlled evidence directory. `npm run collect:release-external-state` remains available for raw evidence capture through `scripts/collect_release_external_state.ps1`, and its `docs/release_external_state_evidence.local.json` output must stay ignored.
- Do not store real keys, Cookie values, Token values, signatures, Authorization headers, or unredacted sensitive fields in attestation files.
- Keep each blocker in `open` status in `docs/release_readiness_status.json` until its acceptance command or evidence has passed; once closed, keep the command evidence and final-head rerun requirement.
- Figma / Canva handoff cannot be screenshots only; it must include accessible source links, Brand Kit, and design token location.
- Formal Codex Security scan cannot be replaced by `verify_high_risk_security.php`, `composer audit`, or `npm audit`.
