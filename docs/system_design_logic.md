# 宿析OS系统构思逻辑

> 核心思想：不要把宿析OS设计成单点工具，而要设计成一个可观测、可纠偏、可复盘的经营反馈系统。

一句话定位：宿析OS 用授权 OTA 可见数据做经营输入，用采集证据、字段目录和质量校验保证口径，用收益/流量/转化/竞争圈/服务质量/广告/商旅诊断生成待确认行动建议，再通过运营执行和效果复盘沉淀为投资与扩张判断依据。

## 1. 系统定义

宿析OS的系统结构由三部分组成：

| 组成 | 在宿析OS中的含义 |
|---|---|
| 要素 | 授权 OTA 可见数据、日报经营数据、竞争圈/同圈数据、价格、房态、订单、服务质量、执行动作、投资测算记录。 |
| 连接关系 | 采集证据、字段目录、质量校验、指标计算、收益诊断、待确认 AI 建议、审批执行、动作追踪、效果复盘。 |
| 目标 | 通过线上OTA数据驱动收益分析、AI决策、运营管理和投资决策，提高经营质量和决策确定性。 |

系统第一阶段只围绕一个主闭环建设：

```text
授权OTA可见数据 -> 采集证据 -> 字段目录 -> 标准事实 -> 收益/流量/转化/竞争圈诊断 -> 待确认 AI 建议 -> 运营动作 -> 效果复盘 -> 新一轮OTA数据
```

投资决策不是独立孤岛，而是使用经营闭环沉淀的数据、假设、风险和复盘结果。

## 2. 字段到决策的路径

系统文案和页面逻辑必须先说明数据从哪里来、如何被标准化、能支持什么判断，避免把“能看到的数据”直接写成“已闭环能力”。

| 层级 | 说明 | 系统要求 |
|---|---|---|
| 原始来源 | 携程、美团、携程商旅等授权后台页面、接口响应、导出文件或人工录入。 | 保留平台、酒店、日期、采集方式、源更新时间和失败原因。 |
| 字段目录 | 用平台页面和 i18n 语言包中的既有术语统一字段名，例如预订订单数、预订销售额、在店间夜、列表页曝光量、详情页访客量、订单页访客量、订单提交人数、PSI 服务质量分；i18n 只作命名参考，不提供业务值。 | 无法确认含义时保留接口字段名和来源路径，不自行改写口径。 |
| 标准事实 | 把订单、间夜、销售额、曝光、访客、转化、竞争圈、服务质量、广告和商旅等字段沉淀为可分析事实。 | 非数值事实只进 `raw_data`，不伪造金额、间夜或订单数。 |
| 指标诊断 | 按收益、流量、转化、竞争圈、服务质量、广告和商旅拆分原因。 | OTA 口径、竞争圈口径、商旅口径和全酒店口径必须分开。 |
| AI建议 | 基于已验证字段生成待确认行动建议。 | 必须附带证据、影响指标、执行条件、风险和复盘周期。 |
| 运营复盘 | 用下一轮数据验证动作是否有效。 | 记录执行前值、目标值、执行后值和失败原因。 |

## 2.1 i18n术语进入系统的规则

携程后台 i18n 语言包提供字段命名和页面语义参考，但不直接等同于可入库字段，也不等同于业务数据。它包含功能、按钮、指标、提示语、节假日、国家地区等中文资源，部分文件还会带有前端埋点上报代码；这些内容只能帮助定位页面含义和触发线索，不能直接生成订单、流量、收益、竞争圈或服务质量事实。

宿析OS采用以下顺序确认术语：

```text
页面展示名 / i18n术语
-> Request URL / Payload / Preview 或 Response
-> source path 与字段值
-> 标准字段名、中文名、口径和数据状态
```

落地规则：

1. 收益和业绩类字段优先使用“预订订单量、预订销售额、间夜量、在店间夜、平均卖价、出租率、紧张度”等平台术语。
2. 流量漏斗类字段优先使用“列表页曝光量、详情页访客量、订单页访客量、订单提交人数、下单转化率”等平台术语。
3. 竞争圈相关字段只写成“竞争圈平均、竞争圈排名、同圈对比、流失订单及去向”，不写成市场全量。
4. 服务质量相关字段使用“PSI服务质量分、基础分、奖励分、减分项、回复率、点评分”等术语；点评明文明细不进入默认采集和报告。
5. 金字塔推广、携程商旅、BPI等字段必须保留独立渠道口径，不并入普通 OTA 经营结论。

详细术语映射见 `docs/ota_i18n_terminology_logic.md`。

## 2.2 术语到业务对象到动作

平台页面术语进入宿析OS后，不能直接变成结论；必须先落到业务对象，再进入可复盘动作。

| i18n/页面术语 | 业务对象 | 诊断逻辑 | 动作边界 |
|---|---|---|---|
| 预订订单量、预订销售额、间夜量、平均卖价 | OTA收益事实 | 判断订单、间夜、价格和库存压力对收入的贡献 | 只能生成待确认调价、促销或房态建议 |
| 列表页曝光量、详情页访客量、订单页访客量、订单提交人数 | OTA流量漏斗 | 按曝光、点击、下单、成交拆分断点 | 不能用单个转化率直接给执行结论 |
| 竞争圈平均、竞争圈排名、流失订单去向 | 竞争圈对比 | 判断本店在平台同圈中的相对位置 | 竞争圈指标只代表平台定义的同圈对比，不代表市场全量 |
| PSI服务质量分、回复率、点评分、用户画像 | 服务质量与客群结构 | 判断服务、回复、评分和客群是否影响转化 | 点评明文明细不进入默认采集和报告 |
| 曝光、点击、花费、收益、ROAS | 广告投放事实 | 判断投放成本、收益和有效订单质量 | 不把花费或曝光单独写成投放成功 |
| BPI分、商旅经营报告、协议酒店 | 商旅渠道事实 | 单独判断企业客户和协议渠道表现 | 不并入普通 OTA 零售渠道结论 |

## 3. 存量与流量

系统设计必须区分存量和流量，避免只看短期变化。

| 类型 | 示例 | 设计要求 |
|---|---|---|
| 存量 | 酒店评分、历史收入、累计订单、房型资产、客户信任、执行记录、投资项目档案。 | 保留历史、来源、口径、归属酒店和更新时间。 |
| 流量 | 每日订单、曝光、访客、间夜、收入、调价动作、促销动作、投诉、点评新增。 | 标记采集时间、统计周期、平台来源和异常状态。 |

收益诊断不能只解释当天流量波动，还要说明对存量的影响，例如评分、价格带、复购基础、品牌信任和投资判断。

## 4. 反馈回路

宿析OS的核心不是“给建议”，而是形成反馈回路。

```text
数据输入
-> 指标计算
-> 诊断与根因定位
-> AI建议
-> 人工确认或审批
-> 运营动作
-> 效果观察
-> 策略沉淀
```

其中字段口径必须先于 AI 结论：没有 Request URL、Payload、Preview / Response 或字段含义证据时，只能标记为待补证据或 P3 候选，不能进入自动建议和执行闭环。

每个建议必须能回答：

1. 它基于哪些数据和口径。
2. 它要改变哪个指标或存量。
3. 它需要什么执行条件。
4. 它多久后复盘。
5. 它成功、接近成功或失败的判定标准是什么。

## 5. 时间延迟

经营动作不会立即反映为结果。系统必须显式记录观察期，避免频繁误判。

| 场景 | 延迟风险 | 系统要求 |
|---|---|---|
| 调价 | 订单变化可能滞后于曝光和访客变化。 | 记录执行时间、价格生效时间、观察期和复盘窗口。 |
| OTA流量 | 曝光、访客、转化率可能存在平台跨日统计延迟。 | 不把暂未返回或跨日窗口中的流量字段直接当作0。 |
| 点评与服务质量 | 服务动作对评分和转化的影响通常滞后。 | 保留动作记录和后续点评、评分变化。 |
| 投资测算 | 市场验证和回本周期需要长期数据。 | 区分假设、已验证数据和待补证据。 |

## 6. 目标函数

系统不能只优化单一短期指标。不同模块必须明确自己的目标函数。

| 模块 | 优先目标 | 约束 |
|---|---|---|
| OTA数据 | 可采集、可校验、可追溯。 | 不隐藏失败、不用兜底逻辑掩盖缺字段。 |
| 收益分析 | 解释收入、订单、间夜、ADR、RevPAR、转化等变化。 | OTA口径和全酒店口径必须分开。 |
| AI决策 | 输出有证据、有风险、有执行条件的建议。 | 不把建议等同于已执行。 |
| 运营管理 | 把建议变成可审批、可执行、可追踪、可复盘的动作。 | 缺授权、缺映射、缺证据时必须显示阻塞。 |
| 投资决策 | 用经营数据、市场假设和风险复核支持投资判断。 | 不把未验证假设写成事实。 |

## 7. 高杠杆点

宿析OS优先改系统结构，而不是只堆功能。

| 杠杆点 | 落地方式 |
|---|---|
| 信息流 | 每个指标显示来源、口径、时间、状态。 |
| 反馈速度 | 缩短采集、诊断、动作、复盘之间的链路。 |
| 决策权 | AI只生成建议，关键动作需要审批或明确的人机协同规则。 |
| 目标函数 | 从“订单量”升级为收入质量、利润、评分、转化、现金流和投资回报的组合判断。 |
| 结构约束 | 缺字段、缺授权、缺映射、缺证据时进入阻塞状态，而不是假成功。 |

## 8. 设计原则

1. 先画清楚输入、标准化、诊断、决策、执行、复盘和沉淀，再设计页面和接口。
2. OTA数据默认是渠道口径，未接入PMS、CRS、全量房量和全量收入前，不写成全酒店经营口径。
3. AI建议必须绑定证据、目标指标、预期变化、风险和复盘周期。
4. 运营动作必须保留执行前值、目标值、执行后值、执行证据和失败原因。
5. 投资决策必须区分真实数据、模型推演、人工假设和待补证据。
6. 系统宁可暴露“不可判断”或“已阻塞”，也不要用默认值、静默catch或假成功掩盖问题。

## 9. AI性能与稳定性

AI不是单独的“智能层”，而是经营反馈系统中的诊断与建议生成环节。性能和稳定性也必须按系统结构设计：

| 维度 | 原则 |
|---|---|
| 性能 | 少生成、少请求、少传无关上下文；能用规则解决的指标计算、状态判断和权限校验不用 LLM。 |
| 稳定 | 结构化输出、显式错误、有限重试、低置信度标记、人工确认和评估集回归必须形成闭环。 |
| 可观测 | 每次影响运营或投决的 AI 调用必须能追踪模型、Prompt 版本、输入口径、来源引用、耗时、状态和人工确认结果。 |
| 可复盘 | AI 输出必须进入动作追踪和效果复盘；失败样本沉淀为评估集，而不是只修改提示词。 |

详细复盘和落地清单见 `docs/system_beauty_ai_stability_review.md`。
