# 宿析OS系统构思逻辑

> 核心思想：不要把宿析OS设计成单点工具，而要设计成一个可观测、可纠偏、可复盘的经营反馈系统。

## 1. 系统定义

宿析OS的系统结构由三部分组成：

| 组成 | 在宿析OS中的含义 |
|---|---|
| 要素 | OTA数据、日报经营数据、竞对数据、价格、房态、订单、点评、执行动作、投资测算记录。 |
| 连接关系 | 数据采集、指标计算、收益诊断、AI建议、审批执行、动作追踪、效果复盘。 |
| 目标 | 通过线上OTA数据驱动收益分析、AI决策、运营管理和投资决策，提高经营质量和决策确定性。 |

系统第一阶段只围绕一个主闭环建设：

```text
OTA数据 -> 收益分析 -> AI决策建议 -> 运营管理 -> 效果复盘 -> 新一轮OTA数据
```

投资决策不是独立孤岛，而是使用经营闭环沉淀的数据、假设、风险和复盘结果。

## 2. 存量与流量

系统设计必须区分存量和流量，避免只看短期变化。

| 类型 | 示例 | 设计要求 |
|---|---|---|
| 存量 | 酒店评分、历史收入、累计订单、房型资产、客户信任、执行记录、投资项目档案。 | 保留历史、来源、口径、归属酒店和更新时间。 |
| 流量 | 每日订单、曝光、访客、间夜、收入、调价动作、促销动作、投诉、点评新增。 | 标记采集时间、统计周期、平台来源和异常状态。 |

收益诊断不能只解释当天流量波动，还要说明对存量的影响，例如评分、价格带、复购基础、品牌信任和投资判断。

## 3. 反馈回路

宿析OS的核心不是“给建议”，而是形成反馈回路。

```text
数据输入
-> 指标计算
-> 诊断与根因定位
-> AI建议
-> 人工确认或审批
-> 运营动作
-> 效果观察
-> 策略沉淀
```

每个建议必须能回答：

1. 它基于哪些数据和口径。
2. 它要改变哪个指标或存量。
3. 它需要什么执行条件。
4. 它多久后复盘。
5. 它成功、接近成功或失败的判定标准是什么。

## 4. 时间延迟

经营动作不会立即反映为结果。系统必须显式记录观察期，避免频繁误判。

| 场景 | 延迟风险 | 系统要求 |
|---|---|---|
| 调价 | 订单变化可能滞后于曝光和访客变化。 | 记录执行时间、价格生效时间、观察期和复盘窗口。 |
| OTA流量 | 曝光、访客、转化率可能存在平台跨日统计延迟。 | 不把暂未返回或跨日窗口中的流量字段直接当作0。 |
| 点评与服务质量 | 服务动作对评分和转化的影响通常滞后。 | 保留动作记录和后续点评、评分变化。 |
| 投资测算 | 市场验证和回本周期需要长期数据。 | 区分假设、已验证数据和待补证据。 |

## 5. 目标函数

系统不能只优化单一短期指标。不同模块必须明确自己的目标函数。

| 模块 | 优先目标 | 约束 |
|---|---|---|
| OTA数据 | 可采集、可校验、可追溯。 | 不隐藏失败、不用兜底逻辑掩盖缺字段。 |
| 收益分析 | 解释收入、订单、间夜、ADR、RevPAR、转化等变化。 | OTA口径和全酒店口径必须分开。 |
| AI决策 | 输出有证据、有风险、有执行条件的建议。 | 不把建议等同于已执行。 |
| 运营管理 | 把建议变成可审批、可执行、可追踪、可复盘的动作。 | 缺授权、缺映射、缺证据时必须显示阻塞。 |
| 投资决策 | 用经营数据、市场假设和风险复核支持投资判断。 | 不把未验证假设写成事实。 |

## 6. 高杠杆点

宿析OS优先改系统结构，而不是只堆功能。

| 杠杆点 | 落地方式 |
|---|---|
| 信息流 | 每个指标显示来源、口径、时间、状态。 |
| 反馈速度 | 缩短采集、诊断、动作、复盘之间的链路。 |
| 决策权 | AI只生成建议，关键动作需要审批或明确的人机协同规则。 |
| 目标函数 | 从“订单量”升级为收入质量、利润、评分、转化、现金流和投资回报的组合判断。 |
| 结构约束 | 缺字段、缺授权、缺映射、缺证据时进入阻塞状态，而不是假成功。 |

## 7. 设计原则

1. 先画清楚输入、存量、流量、反馈、延迟、目标，再设计页面和接口。
2. OTA数据默认是渠道口径，未接入PMS、CRS、全量房量和全量收入前，不写成全酒店经营口径。
3. AI建议必须绑定证据、目标指标、预期变化、风险和复盘周期。
4. 运营动作必须保留执行前值、目标值、执行后值、执行证据和失败原因。
5. 投资决策必须区分真实数据、模型推演、人工假设和待补证据。
6. 系统宁可暴露“不可判断”或“已阻塞”，也不要用默认值、静默catch或假成功掩盖问题。
