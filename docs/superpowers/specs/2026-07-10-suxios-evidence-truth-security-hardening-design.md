# 宿析OS证据真实性与安全加固设计

日期：2026-07-10
状态：已确认方案 A，待实施计划
范围：`HOTEL/`

## 1. 目标

在保持现有业务入口和旧数据可迁移的前提下，修复以下已确认问题：

1. OTA Cookie、Token、签名与授权材料不得以明文进入通用配置、通用缓存或通用配置响应。
2. OTA 配置访问必须同时受酒店权限范围约束，不能仅凭配置创建人获得跨酒店访问权。
3. 未确认的 OTA 渠道策略不得被自动解释为双渠道。
4. 未采集、缺字段、不可计算和真实零值必须可区分。
5. OTA 日事实写入必须具备数据库级幂等约束。
6. 前端不得把失败、部分成功、过期、对象存在或页面访问历史解释为证据已就绪。
7. 首页、运营和投资页面必须读取同一份按酒店与经营日键控的闭环状态。
8. 发布门禁必须区分静态合同、运行就绪和真实浏览器业务链验证。
9. 在不启动整体前端重写的前提下，降低登录与首屏的全量脚本解析负担。

## 2. 业务边界

- OTA 渠道证据只代表携程或美团渠道，不升级为全酒店经营事实。
- 收益分析只能消费状态为 `verified` 且未过期的适用渠道证据。
- AI 决策必须继承输入证据的渠道范围、经营日期和缺口状态。
- 运营执行必须经过人工审批、执行证据和复盘状态，不以“存在记录”代替完成。
- 投资判断必须通过 `operating_data_gate`，不以页面类型或转让快照存在代替经营闭环。
- 不新增真实 OTA 自动写价、自动确认或自动执行能力。
- 不引入 Vite 或新前端框架，不重写无关页面。

## 3. 总体架构

方案分为五个可独立验收的阶段。每一阶段先写失败测试，再做最小实现；上游门禁未通过时，下游保持显式阻断。

### 3.1 凭据保险库与租户边界

新增 `ota_credentials` 表，仅保存加密后的敏感载荷和必要索引信息：

- `platform`
- `config_id`
- `system_hotel_id`
- `encrypted_payload`
- `payload_version`
- `secret_mask`
- `credential_status`
- `created_by`
- `rotated_at`
- `create_time` / `update_time`

唯一约束为 `platform + config_id`。酒店ID用于权限校验和迁移审计，不允许调用方通过配置ID绕过酒店范围。

新增独立加密服务：

- 使用带认证的 AES-256-GCM；密文包含版本、随机 nonce、认证标签和 ciphertext。
- 密钥来自新的 `OTA_CREDENTIAL_SECRET` 环境变量，不进入数据库、日志、响应或仓库。
- 缺少密钥、密文认证失败或载荷版本未知时返回明确错误，不回退读取明文。
- 加密服务与现有 AI Key 的 CBC 格式隔离，不在本项目中顺带迁移 AI 配置。

`ctrip_config_list` 与 `meituan_config_list` 只保留非敏感元数据、酒店绑定、状态和 `credential_ref`。通用配置读取接口对所有敏感键执行拒绝或结构化脱敏，超级管理员也不获得原始密文或明文。

一次性迁移流程：

1. dry-run 盘点明文配置、缺酒店绑定、重复配置ID和无法识别载荷。
2. 只有 `OTA_CREDENTIAL_SECRET` 有效且酒店绑定明确时才写入保险库。
3. 在同一事务中写入加密记录并从通用配置移除明文字段。
4. 清理应用内缓存和持久配置缓存。
5. 输出不含秘密的迁移结果及需要人工处理的阻断项。
6. 迁移完成后轮换或失效旧凭据；轮换证明仍作为独立发布证据。

现有未提交的酒店权限收紧补丁保留并纳入验证。迁移前先盘点未绑定酒店的旧配置，避免收紧后形成不可见孤儿记录。

### 3.2 渠道策略、缺失状态与事实幂等

`hotels.ota_channel_strategy` 扩展为：

- `unconfirmed`
- `ctrip_only`
- `dual`
- `meituan_only`

新建门店默认 `unconfirmed`。旧空值或非法值迁移为 `unconfirmed`，不迁移为 `dual`。界面统一显示“未确认 / 仅携程 / 携程+美团 / 仅美团”。确认信息记录确认人、确认时间和来源。

指标采用“值 + 状态 + 来源”合同：

- `not_collected`
- `missing_field`
- `not_calculable`
- `zero_observed`
- `verified`
- `partial`
- `stale`
- `failed`
- `policy_disabled`

真实数值可以为零，但只有 `zero_observed` 或 `verified` 才能把零显示为经营事实。缺失分母返回 `null + not_calculable`。

为 `online_daily_data` 增加稳定 `fact_key` 和唯一索引。事实键由酒店、平台、经营日期、数据类型、维度、比较口径、数据期间或源记录标识规范化生成。实时快照必须包含快照桶或源记录ID，避免把不同时间点误合并。写入改为原子 upsert；迁移前先生成重复组报告，再按确定规则合并，不静默删除。

### 3.3 权威闭环状态与前端状态机

新增按 `hotel_id + business_date` 查询的闭环总览合同，分开返回：

- `ota_evidence_ready`
- `revenue_ready`
- `ai_decision_ready`
- `approved`
- `executed`
- `execution_evidence_ready`
- `reviewed`
- `roi_ready`
- `investment_judgement_allowed`

每个字段同时提供 `status`、`reason_code`、`source_scope`、`source_date` 和 `updated_at`。首页、运营跟踪和 P4 投资页面只消费该合同，不读取其他页面是否访问过或局部 ref 是否已有数据。

前端规则：

- 只使用正向状态白名单；`partial/stale/failed/not_collected/policy_disabled` 不得显示绿色就绪。
- 完整诊断使用 `idle/loading/success/partial/error`，不得在 `finally` 中写成功。
- 页面加载去重器只缓存 fulfilled；rejected 必须清理进行中状态并继续抛错。
- 所有按门店请求捕获 `hotel_id` 和请求序号；响应写入前再次校验，旧门店响应直接丢弃并保留可诊断状态。
- P4 “已闭合经营数据”绑定真实投资准入门；刷新失败时保留旧值但标记 `stale` 和上次成功时间。
- 携程概况采集区分 `parsed/partial/empty/failed`，`empty` 不生成绿色指标卡。

### 3.4 运行就绪与发布门禁

保留 `/api/health` 作为进程存活检查，新增 `/api/ready`，验证：

- 数据库连接
- 必需迁移版本
- 核心表存在
- OTA 凭据加密配置可用但不回显秘密
- 关键缓存服务可用性

将 `verify:e2e-contracts` 重新定位为静态合同门禁。CI 增加只读运行时 smoke 和 Playwright 业务链：登录、门店隔离、OTA 状态、收益阻断、AI 阻断、运营状态、P4 投资阻断。测试不得触发真实 OTA 写操作。

### 3.5 前端负载治理

不启动整体重构。先将登录和鉴权前入口保持为最小同步资源，认证后按页面加载现有 `*-static.js` 业务域。沿用当前静态 helper 模式，逐域迁移：数据健康、携程、美团、收益AI、系统管理。

增加静态预算：

- 登录首屏同步业务脚本字节数
- 初始请求数量
- `public/index.html` 体积趋势
- 单个静态 helper 体积上限

预算超限时 CI 失败，但不以静态预算代替真实浏览器指标。

## 4. 错误处理与兼容

- 旧客户端未提交渠道策略时保留已确认值；没有已确认值时保持 `unconfirmed`。
- 旧明文凭据只允许由显式迁移命令读取，不允许正常请求路径兼容读取。
- 迁移未完成的配置返回 `migration_required`，不伪装为 `login_required` 或 `missing_cookie`。
- 旧指标响应在迁移期可保留数值字段，但必须增加状态字段；前端优先读取状态，没有状态时显示“状态未确认”而不是成功。
- 数据库迁移必须具备可重复执行能力；结构迁移和数据迁移分开。
- 任何失败均保留原因码，不使用宽泛 catch 或默认零隐藏。

## 5. 测试策略

每个阶段遵循 RED → GREEN → REFACTOR：

1. 凭据：数据库、通用配置响应、缓存和日志中不得出现测试秘密；密文篡改必须解密失败。
2. 租户：配置创建人失去酒店范围后不可见、不可修改；超级管理员仍按明确管理权限操作。
3. 渠道：空值和非法值保持 `unconfirmed`；仅携程和仅美团分母对称。
4. 指标：缺字段、真实零、缺分母、部分解析分别断言。
5. 幂等：同一事实重复与并发写入只保留一条；不同快照不误合并。
6. 前端：失败不显示成功、失败加载可重试、跨门店慢响应不串写、P4 未准入不显示闭合。
7. 运行门禁：数据库断开时 `/api/health` 仍可存活，但 `/api/ready` 必须失败。
8. 浏览器：只读业务链可运行，真实 OTA 外部写操作保持禁用。

阶段验证通过后再运行 PHPUnit、Node 合同测试、P0 guards、只读 Playwright smoke 和发布状态审查。

## 6. 发布与回滚

发布顺序：

1. 部署结构迁移和加密密钥。
2. 执行凭据 dry-run，人工处理缺酒店绑定项。
3. 执行凭据迁移与旧凭据轮换。
4. 部署只读新合同和兼容响应。
5. 切换前端消费新状态。
6. 启用新运行门禁。
7. 最后启用首屏按域加载。

回滚不得重新写回明文。应用回滚时保留加密表和新状态字段；必要时回滚到只读阻断状态，而不是恢复不安全兼容路径。

## 7. 验收标准

- 通用配置表、通用缓存和通用配置响应不再包含 OTA 明文秘密。
- 任意酒店范围外用户无法读取或维护 OTA 配置。
- 未确认渠道不计入双渠道就绪率。
- 缺失值不显示为 0，缺分母不显示为 0%。
- 数据库唯一约束证明重复事实不可插入。
- 首页结果不受页面访问顺序影响，切换门店不会写入旧响应。
- 诊断失败、零行采集和 P4 未准入不会显示绿色成功或已闭合。
- `/api/ready` 能识别数据库或迁移未就绪。
- CI 同时具备静态合同、运行时 smoke 和只读浏览器业务链证据。
- 前端首屏资源预算有自动门禁，并以浏览器实测补充验证。

## 8. 非目标

- 不新增或放宽 OTA 平台抓取权限。
- 不绕过登录、验证码、人机验证或平台授权。
- 不自动写入 OTA 价格、库存或运营动作。
- 不把 OTA 渠道数据扩展成全酒店经营事实。
- 不在本轮迁移 AI 模型密钥格式。
- 不进行全站视觉改版或前端框架替换。
