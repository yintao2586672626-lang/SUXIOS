# 携程默认全量授权与房量配置设计

## 目标

简化门店携程配置：门店不再选择 Profile 采集范围，保存携程配置后默认具备全部可采模块权限；同时录入酒店实际房量和竞争圈总房量，为携程渠道分析提供明确容量参数。

## 业务口径

- `hotel_room_count`：当前门店固定的实际总客房数，必须为正整数。
- `competitor_room_count`：携程当前竞争圈所覆盖酒店的总房量，必须为正整数。
- 两个字段属于携程配置元数据，不是 Cookie、Token 或其他凭据。
- 两个字段只用于携程 OTA 渠道容量与竞争圈对比，不等于每日可售房量，也不代表全酒店经营事实。
- 每日可售房量仍由 OTA 采集或经营日报提供，缺失时不得用固定总房量冒充。

## 采集授权设计

- 前端删除“Profile采集范围”可编辑输入框，不再让门店选择 `default`、`core`、`wide` 或自定义模块。
- 携程配置新建、编辑和健康面板快速保存时，统一保存 `capture_sections = all`。
- 后端是最终权威边界：即使旧客户端传入较小范围，普通携程配置保存也会归一为 `all`。
- `all` 表示该配置允许所有已支持的携程模块。具体运行仍可由内部任务按目标模块执行子集，例如只跑点评或流量，避免日常任务无必要地全量执行。
- 不改变门店、租户和携程平台账号的绑定边界；“全部授权”不得跨门店或跨租户复用 Profile/Cookie。

## 界面与数据流

### 新建/编辑携程配置

表单保留系统门店、携程酒店 ID 和 Cookie/API 辅助内容，删除 Profile 采集范围，新增：

1. 酒店实际房量：数字输入，正整数，必填。
2. 竞争圈总房量：数字输入，正整数，必填。

前端负责即时提示，后端再次校验。保存成功后返回脱敏配置元数据，列表和编辑表单回显两个房量。

### 旧配置

- 旧配置没有房量时仍可读取和用于原有采集，不伪造默认房量。
- 界面显示“房量待补”，不把缺失值显示为 `0`。
- 用户再次保存旧配置时，必须补齐两个房量。
- 旧配置的有效采集权限按 `all` 解释，不需要门店手工迁移 Profile 范围。

## 存储

- 复用现有 `system_configs.ctrip_config_list` 配置元数据结构，新增 `hotel_room_count` 和 `competitor_room_count`。
- 不新增数据库表，不修改凭据 Vault 结构。
- 房量保持在非敏感元数据中，可正常用于列表回显和后续渠道分析。

## 错误处理

- 任一房量为空、零、负数、小数或非数字时，保存返回 `422` 及字段级中文提示。
- 房量校验失败不得覆盖旧配置。
- 不强制两个房量的大小关系；竞争圈可能按平台当前口径排除或纳入本店，系统只保留用户确认的正整数事实。

## 验收与测试

1. 携程配置界面不再出现 Profile 采集范围输入框。
2. 新建配置传入较小范围时，后端持久化结果仍为 `capture_sections = all`。
3. 前端保存请求携带两个房量，后端保存并返回脱敏元数据。
4. 编辑配置时两个房量正确回显。
5. 缺失或非法房量无法保存，且不改动旧配置。
6. 旧配置缺房量时显示待补状态，原有采集仍可执行。
7. 内部点评、流量等专项任务仍可传入任务级模块子集。
8. 运行定向 PHP/Node 合同测试、PHP 语法检查和前端入口校验。

## 非目标

- 不修改美团配置房量口径。
- 不把固定总房量自动写入每日可售房量。
- 不变更携程登录、验证码、门店绑定、租户隔离或凭据安全边界。
- 不要求日常采集每次实际运行所有模块。
