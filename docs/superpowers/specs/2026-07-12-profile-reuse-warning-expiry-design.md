# OTA Profile 持久复用、提醒与强制续登设计

## 目标

携程和美团浏览器 Profile 在一次真实登录并完成门店、平台、数据源绑定后持续复用，不再要求每天重新授权。Profile 只在平台实际判定登录失效，或距离最后一次真实登录达到强制续登期限时阻断自动采集。

## 复用规则

- 登录后第 0–6 天：直接自动采集，不提示续登。
- 登录后第 7–9 天：仍允许自动采集，但账户状态中心提示“建议重新登录，剩余 N 天”。
- 登录后第 10 天起：停止自动采集，状态改为“登录已到期”，要求账号使用者重新登录。
- 任意时间出现平台跳回登录页、`login_required`、`session_expired`、401 或 403 等明确登录失效证据：立即停止复用并要求重新登录，不等待第 10 天。
- 新一次真实登录验证成功后，复用天数从该次 `current_session_probe_at` 重新计算。

## 状态模型

新增独立的 Profile 复用判断，不改变现有当天证据判断：

- `reusable`：登录年龄小于 7 天，可自动采集。
- `renewal_warning`：登录年龄达到 7 天但小于 10 天，可自动采集并提示续登。
- `expired`：登录年龄达到 10 天或已有明确登录失效证据，禁止自动采集。
- `unverified`：缺少真实登录证明、门店绑定、平台身份或 Profile 数据源，禁止自动采集。

`isCurrentVerified()` 继续用于当天 OTA 证据、P0 和下游经营判断，不放宽为历史登录证明；自动采集入口改用新的 Profile 复用状态，避免把“可打开平台采集”误写成“当天数据已闭环”。

## 数据流

1. 用户完成一次真实平台登录，系统保存 Profile 和完整作用域证明。
2. 自动采集前读取同一数据源的复用状态。
3. `reusable` 或 `renewal_warning` 进入原有采集脚本；采集脚本继续执行真实页面登录检查。
4. 采集成功按原流程保存和回读数据；`renewal_warning` 不阻断入库。
5. 采集明确返回登录失效时，数据源保留失败原因，账户中心立即显示重新登录入口。
6. `expired` 或 `unverified` 在启动采集前阻断，不弹出服务器端登录窗口。

## 兼容与边界

- 不读取、展示或迁移 Cookie、令牌、密码等凭据内容。
- 继续要求 Profile 数据源启用、门店/租户/平台/Profile 标识绑定一致。
- 现有具备完整登录证明的 Profile 可直接按最后登录时间计算，无需重新登录迁移。
- 仅改变携程/美团 Profile 自动采集可用性；不改变 Cookie/API 手动采集、当天数据质量、收益分析或 AI 决策门禁。

## 验证

- 单元测试覆盖第 6、7、9、10 天边界和提前登录失效。
- 自动采集测试覆盖 `reusable`、`renewal_warning` 可进入采集，`expired`、`unverified` 被阻断。
- 状态接口与页面测试覆盖第 7 天提醒、第 10 天重新登录。
- 对敦煌漠蓝新做只读状态复核，确认昨天验证的携程和美团进入可复用状态，不伪造新的当天采集或入库结果。
