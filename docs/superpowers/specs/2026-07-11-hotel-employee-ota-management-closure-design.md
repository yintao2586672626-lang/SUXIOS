# 酒店、员工与 OTA 配置管理收口设计

日期：2026-07-11
状态：用户已确认

## 目标

在不扩展员工分页、未分配员工处理和复杂酒店合并能力的前提下，收口酒店删除、同名酒店识别、OTA 配置生效状态、负责人填写、员工默认密码及酒店停用说明。

## 已确认行为

### 酒店删除

- 只有超级管理员可以删除酒店，前端不向其他角色展示删除入口，后端继续强制鉴权。
- 删除在一个数据库事务内完成：清除该系统酒店对应的业务数据、OTA 数据、采集记录、平台配置、凭证和 Profile 绑定，再删除酒店记录。
- 员工账号不删除；解除该酒店的员工授权，主门店指向被删酒店的员工改为未分配。
- 保留一条脱敏管理员操作日志，只记录操作者、时间、酒店 ID 和酒店名称，不保留 Cookie、令牌或经营数据。
- 任一步删除失败则整体回滚并返回真实错误，不返回部分成功。

### 同名酒店与合并

- 新增或改名时，由后端按去除首尾空白后的酒店名称检查重复。
- 发现同名酒店时不直接保存，返回现有酒店的 ID、名称、编码和状态。
- 页面提示酒店重复，并为超级管理员提供打开现有轻量数据迁移流程的入口。
- 不扩展酒店合并的数据范围和流程复杂度。

### OTA 配置状态

- `保存成功` 只表示配置和加密凭证已落库。
- 新保存配置先标记为待验证，不因存在 Cookie 自动成为验证成功配置。
- 只有授权登录完成、当前会话证明有效、平台酒店身份与系统酒店匹配后，才标记为验证成功并成为当前配置。
- 同一酒店同一平台以最新验证成功配置为主；旧配置保留为非激活历史，不阻碍新配置保存。
- 页面分别显示“已保存，待验证”和“验证成功，当前使用”，不再使用含混的“最新成功配置”。

### 负责人、密码与停用

- 酒店负责人继续使用可自由填写的姓名文本，不要求绑定员工账号。
- 新增员工和管理员重置员工密码时，默认密码固定为 `666666`。
- 页面统一使用“密码”，删除“临时密码”、随机生成和“换一组”交互；复制内容显示登录地址、用户名和密码。
- 员工自行修改密码仍遵守系统密码策略。
- 停用酒店只停止该酒店访问和 OTA 采集，不停用员工账号；有其他酒店权限的员工仍可登录。

## 非目标

- 不处理当前三个未分配员工。
- 不改员工和酒店列表的全量分页方式。
- 不增加复杂酒店合并能力。
- 不修改高级 AI 工具箱及无关模块。

## 验证

- 后端：酒店删除权限、完整级联清理、事务回滚、同名拦截、员工账号保留、密码 `666666` 创建/重置。
- OTA：保存后为待验证；仅当前会话和酒店身份验证通过后激活；旧配置不再保持当前状态。
- 前端：删除提示、同名提示、配置状态、负责人文案、默认密码和停用说明。
- 运行现有酒店管理、员工管理、权限、OTA 配置和登录态定向测试，并在本地页面刷新检查。

## 风险控制

- 删除属于不可逆操作，执行前展示将删除的数据分类和数量，并要求再次确认酒店名称。
- 删除清单只覆盖数据库中实际存在且含系统酒店关联列的表；凭证和系统配置需通过专用清理逻辑处理。
- 不打印或返回原始 Cookie、令牌和凭证内容。
