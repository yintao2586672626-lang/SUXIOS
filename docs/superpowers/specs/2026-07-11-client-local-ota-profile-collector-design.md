# 账号使用者本机 OTA Profile 采集器设计

> 日期：2026-07-11  
> 已确认方向：账号使用者本机持有 Profile；宿析主机只协调任务、记录状态并接收标准化采集结果。  
> 首个验收对象：系统门店 `80`（敦煌漠蓝新）美团登录。

## 0. 当前实施决策

当前电脑尚未部署独立远程服务器，因此第一阶段把同一台电脑同时视为“宿析服务器”和“账号使用者电脑”，先完成真实可用的本机 Profile 登录闭环：

```text
本机宿析网页
-> 本机登录任务
-> 本机专用 Profile 浏览器
-> 用户人工验证
-> 当前会话证明
-> 本机宿析数据库状态回显
```

第一阶段不建设异地本机采集器、设备配对或远程任务队列，也不确定最终数据部署位置。后续部署中央服务器时，再按本文第 4 至 8 节拆分客户端与服务器职责。

第一阶段业务数据只写当前宿析数据库；本机不额外建立第二份业务数据库。Profile 目录继续独立保存浏览器会话。Profile 目录存在、体积增长或曾启动浏览器，均不得作为登录成功证据，只有当前会话探测通过才能显示“自动可采集”。

## 1. 目标

建立可用于异地账号授权的真实链路：用户在哪台电脑打开宿析并执行 OTA 授权，登录窗口就在哪台电脑弹出，Profile 也只保存在该电脑。宿析服务器不得代替异地用户打开平台浏览器，不接收 OTA 密码、验证码、完整 Cookie 或浏览器 Profile。

这条链路服务于：

```text
账号使用者本机 OTA Profile
-> 当前会话验证
-> 单门店采集
-> 标准化结果上传
-> OTA 渠道收益分析
-> AI 决策与运营执行
```

## 2. 不做的事情

- 不在宿析服务器或管理员电脑弹出异地用户的携程/美团窗口。
- 不要求用户把 OTA 密码、验证码、Cookie 或 Profile 目录转交给管理员。
- 不把“设备在线”“历史登录成功”或“Profile 目录存在”标记为当前自动可采集。
- 首版不做跨电脑 Profile 搬迁；更换电脑时在新电脑重新授权，并撤销旧设备。
- 首版不做 macOS、本机采集器自动更新和浏览器扩展。

## 3. 方案选择

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| 本机采集器 | Profile 和采集都留在用户电脑；可持续自动采集；平台验证码由用户完成 | 需要运行轻量客户端 | 采用 |
| 浏览器扩展 | 可复用用户现有浏览器 | 浏览器兼容、权限、自动采集和版本治理复杂 | 后续可选 |
| 浏览器辅助 JSON | 无需常驻客户端，现有能力可复用 | 必须人工导出/导入，不能稳定自动采集 | 保留为回退 |

## 4. 系统边界

### 4.1 宿析网页

- 展示门店、平台、已绑定设备、设备在线状态、Profile 当前验证状态和最近采集结果。
- 创建带酒店、平台、平台门店和设备范围的短期任务。
- 不直接启动服务器浏览器。
- 当没有可用设备时显示“未绑定本机采集器”，而不是“可采集”。

### 4.2 宿析服务器

- 负责设备配对、任务队列、权限校验、状态汇总、结果验签和入库。
- 只保存设备公钥/标识摘要、Profile 标识摘要、酒店与平台绑定、验证时间、失败代码和采集证据引用。
- 不保存 OTA 密码、验证码、完整 Cookie、Authorization 或浏览器 Profile。

### 4.3 账号使用者本机采集器

- 由账号使用者在自己的电脑运行。
- 与宿析服务器完成一次性设备配对，并持续领取分配给该设备的任务。
- 按 `平台 + 平台门店` 使用独立 Profile 目录。
- 在本机打开可见 Chromium 窗口，让用户完成短信、验证码或人机验证。
- 验证当前会话后，只回传脱敏状态证明；采集时回传标准化业务数据和证据摘要。

## 5. 数据模型

### 5.1 复用现有表

- `ota_profile_bindings`：继续表示 `租户 + 系统门店 + 平台 + Profile 摘要` 的权威绑定。
- `platform_data_sources`：继续表示 OTA 数据源及采集方式；本机采集器数据源使用 `browser_profile`。
- `platform_data_sync_tasks`：继续表示目标日期采集任务和执行结果。
- `OtaProfileSessionProofService`：继续作为当前会话证明的消费边界，禁止用历史元数据替代当前证明。

### 5.2 新增表

#### `ota_client_devices`

- `id`
- `tenant_id`
- `user_id`
- `device_public_id`
- `device_key_hash`
- `device_name`
- `platform`：首版 `windows`
- `status`：`online / offline / revoked`
- `last_seen_at`
- `create_time / update_time`

#### `ota_client_assignments`

- `id`
- `tenant_id`
- `device_id`
- `system_hotel_id`
- `ota_platform`：`ctrip / meituan`
- `profile_key_hash`
- `status`：`active / revoked`
- `assigned_by`
- `create_time / update_time`

设备、酒店、平台和 Profile 摘要必须唯一匹配；一个门店同一平台首版只保留一个活动采集设备。

## 6. 接口与任务流

### 6.1 设备配对

1. 用户在宿析网页生成一次性配对码，作用域包含租户、用户和允许管理的酒店。
2. 本机采集器提交配对码和本机生成的设备公钥。
3. 服务器保存设备摘要并返回短期访问凭据；原始配对码立即失效。
4. 管理员可撤销设备，撤销后停止领取任务和上传结果。

### 6.2 Profile 登录

1. 用户在门店卡片点击“在账号使用者电脑授权”。
2. 网页创建 `profile_login` 任务并指定活动设备；服务器不启动浏览器。
3. 本机采集器通过长轮询领取任务，首版不要求 WebSocket。
4. 本机采集器打开该门店的专用 Profile 浏览器并进入平台 eBooking。
5. 用户完成平台登录。
6. 本机采集器执行轻量登录探测并回传：任务 ID、设备 ID、酒店、平台、Profile 摘要、验证状态、验证时间和失败代码。
7. 服务器验证设备、任务和酒店范围后调用现有会话证明服务更新状态。

### 6.3 数据采集

1. 服务器为已绑定且在线的设备创建单门店目标日期任务。
2. 本机采集器复用本地 Profile 采集携程/美团数据。
3. 本机采集器上传标准化行、数据日期、来源摘要、字段缺口和任务统计，不上传浏览器凭据。
4. 服务器通过现有 `PlatformDataSyncService / OtaBrowserAssistImportService` 边界入库。
5. 只有数据库读回和字段验证通过，页面才显示“自动可采集/本次采集成功”。

## 7. 状态模型

设备状态与登录状态必须分开：

| 状态 | 含义 | 是否自动可采集 |
| --- | --- | --- |
| `client_not_bound` | 门店未绑定本机采集器 | 否 |
| `client_offline` | 已绑定设备当前离线 | 否 |
| `waiting_user_login` | 浏览器已在用户电脑打开，等待人工验证 | 否 |
| `captcha_required` | 需要验证码/短信/人机验证 | 否 |
| `current_session_verified` | 当前设备、当前 Profile 会话已验证 | 是，仍需设备在线 |
| `login_expired` | 会话已失效 | 否 |
| `cookies_incomplete` | 页面可进但业务 Cookie 不完整 | 否 |
| `resource_busy_login` | 同一 Profile 已有登录或采集任务 | 否 |
| `collection_failed` | 登录可用但采集失败 | 否，显示真实原因 |

页面不得继续使用“Profile 目录存在”或旧的 `manual_login_state_verified` 直接推出“登录态已验证”。

## 8. 安全和隐私

- 配对码一次性、短有效期、使用后立即失效。
- 任务必须绑定 `tenant_id + system_hotel_id + platform + device_id`，不允许跨酒店领取。
- 本机采集器上传结果需带设备签名、任务随机数和时间戳，服务器拒绝重放。
- Profile 目录只留在本机；默认不复制、不上传、不打包。
- 日志禁止记录密码、验证码、完整 Cookie、Authorization、手机号和客人隐私。
- 设备撤销后，服务器停止派发任务；是否删除本机 Profile 由账号使用者在该电脑明确执行，服务器不得远程删除。

## 9. UI 设计

门店平台卡片展示四个独立事实：

1. 本机采集器：未绑定 / 在线 / 离线 / 已撤销。
2. Profile：未建立 / 等待登录 / 当前会话已验证 / 已过期。
3. 采集方式：自动采集 / 手动辅助 / 不可采集。
4. 最近采集：目标日期、成功/失败、入库行数和失败原因。

主按钮根据状态变化：

- 无设备：`绑定账号使用者电脑`
- 设备在线未登录：`在该电脑打开美团登录`
- 等待用户：`等待账号使用者验证`
- 已验证：`验证并采集`
- 设备离线：`通知账号使用者启动采集器`

## 10. 首版实施范围

第一阶段仅支持 Windows 与美团，完成后复用同一协议接入携程：

- 一个 Windows 本机采集器进程。
- 一次性设备配对。
- HTTP 长轮询领取任务。
- 美团 `login_only` Profile 登录。
- 当前会话状态回传。
- 单门店采集任务与标准化结果上传。
- 浏览器辅助 JSON 继续作为无客户端回退。

## 11. 漠蓝新美团验收

以系统门店 `80` 为唯一测试门店：

1. 当前电脑作为“账号使用者电脑”运行本机采集器并完成配对。
2. 宿析网页显示该设备在线且只绑定门店 80 美团。
3. 点击授权后，美团窗口只在本机采集器所在电脑弹出；宿析服务器进程树不出现 Chromium 登录窗口。
4. 用户完成人工验证，Profile 保存到本机美团门店专用目录。
5. 服务器只收到脱敏会话证明，状态变为 `current_session_verified`。
6. 重启本机采集器后，无需再次输入密码即可通过轻量登录探测；若平台要求验证则真实显示 `captcha_required / login_expired`。
7. 运行一次单门店美团采集，确认来源、门店、目标日期、入库和数据库读回一致。
8. 停止本机采集器后，网页在心跳超时内变为 `client_offline`，不得继续显示“自动可采集”。

## 12. 失败处理

- 本机采集器未运行：任务保持待领取，页面显示 `client_offline`。
- 设备签名或酒店范围不匹配：服务器拒绝任务和结果，记录脱敏审计。
- 登录超时：状态为 `waiting_user_login_timeout`，不创建成功证明。
- 同一 Profile 并发登录或采集：返回 `resource_busy_login`。
- 上传中断：任务可按同一随机数恢复一次，禁止重复入库。
- Profile 损坏：显示 `profile_corrupted`，由账号使用者在本机重新建立，不在服务器端删除。

## 13. 停止条件

完成漠蓝新美团的设备配对、用户本机弹窗、Profile 保存、当前会话证明和一次真实采集读回后停止。携程接入、浏览器扩展、自动更新、跨设备迁移和批量部署均需要后续明确任务。
