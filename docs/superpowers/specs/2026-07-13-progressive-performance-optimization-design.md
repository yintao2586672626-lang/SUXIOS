# 宿析OS渐进式性能优化设计

## 目标

在不改变现有页面结构、视觉、菜单、页面入口、业务功能、接口契约、OTA 数据口径和数据库结构的前提下，降低首屏传输、HTML 解析、JavaScript 阻塞、重复资源下载、页面切换重复请求和后端热接口延迟。

本设计采用可回退的分批优化。每一批必须先建立失败测试，再实施最小改动，最后通过结构、业务和浏览器回归；任何一批未验证时，不进入下一批。

## 当前证据基线

- `public/index.html`：3,550,704 bytes。
- 入口包含 16 个 `<script>` 块，其中主 Vue 启动脚本内联体约 1,713,149 bytes。
- 将主启动脚本原样外置后，`index.html` 预计约 1,837,570 bytes，入口 HTML 体积降低约 48%。
- 当前首屏同步引用 13 个本地 JavaScript 资源，另有 3 个本地样式资源。
- 当前本地静态资源已经具备 ETag、immutable 缓存和 gzip 缓存；优化重点不是重复实现缓存，而是让大型启动代码能够独立缓存并与 HTML 解析并行。
- `public/index.html` 仍承载约 42 个页面入口；现有代码已经对部分扩展面板、通知、测试 ID、运营模块和表单辅助做了延迟加载，后续优化必须保留这些机制。

以上数字是当前工作树的静态证据。FCP、LCP、Long Task、登录后接口瀑布和生产数据库 P95 仍需浏览器与接口基准补齐。

## 不变约束

### 页面与交互

- 保留 `#app` 下现有 HTML 模板和 DOM 层级。
- 保留所有 `currentPage`、菜单 path、tab key、`data-testid`、弹窗、toast 和 teleport 边界。
- 不重命名现有 Vue ref、computed、method、事件处理器或全局静态 helper API。
- 不改变 CSS class、页面布局、颜色、间距或组件视觉。
- 正常加载路径的页面内容和操作顺序必须与优化前一致。

### 业务与数据

- 不修改路由、请求参数、响应结构或鉴权行为。
- 不修改携程、美团采集、保存、回读、状态判定或证据口径。
- 不把 OTA 渠道数据扩大为全酒店数据。
- 不修改数据库表结构，不以缓存值掩盖采集失败或字段缺失。

### 工程边界

- 不引入 Vite，不在 `public/` 运行前端构建。
- 不增加新的运行时第三方依赖。
- 保留现有 PHP 内置服务器和 Apache/Nginx 静态资源行为。
- 每批改动必须能够独立回退。

## 方案比较

### 方案 A：渐进式交付层优化，采用

先把主内联启动代码原样外置为版本化静态资源，再按真实基准逐步减少首屏同步 helper 和重复请求。

优点：HTML 模板和业务代码语义不变；第一批即可获得独立缓存和并行解析收益；风险可被现有入口保护与 E2E 覆盖。

代价：首次访问的 JavaScript 总量不会仅因外置而显著下降，后续仍需逐页懒加载和 CSS 优化。

### 方案 B：立即大规模懒加载和 Tailwind 裁剪，不采用

一次性把 OTA、收益 AI、管理端 helper 改为按页面加载，并裁剪 Tailwind 未使用选择器。

优点：理论首屏体积下降更多。

风险：动态 class、同步 helper 绑定和巨型 setup 函数之间存在依赖；未经分批验证容易造成局部页面白屏、样式丢失或首次进入功能不可用。

### 方案 C：重建 Vite 前端，不采用

长期可获得标准代码分包和构建产物，但会改变当前工程入口、开发方式和发布链路，超出本次零破坏优化边界。

## 架构设计

### 第一批：入口脚本外置与并行加载

1. 保留两个短小的 head 内联启动脚本：登录背景预加载和 Font Awesome 延迟加载。
2. 将当前最大的内联 Vue 启动脚本逐字节机械提取到 `public/app-main.js`。
3. 在原脚本位置保留版本化 `app-main.js` 引用；主 HTML 模板不移动。
4. 核心 helper 与 `app-main.js` 使用 `defer`，按原出现顺序执行。
5. `app-main.js` 版本包含内容哈希，入口保护器校验哈希与文件内容一致，避免 immutable 缓存返回旧代码。
6. 增加独立于 `app-main.js` 的资源加载失败表面；文件缺失、网络错误或脚本解析失败时，解除 `v-cloak` 并显示明确错误，不静默白屏。

加载顺序保持为：Vue runtime → shared components → 当前同步 core helpers → `app-main.js` → Vue mount。

### 第二批：首屏 helper 分层

浏览器基线确认第一批稳定后，根据页面真实调用关系把同步 helper 分为三组：

- shell 必需：Vue、系统基础 helper、默认首页必需 helper。
- 登录后默认页必需：首页和罗盘相关 helper。
- 页面专属：携程、美团、数据健康、收益 AI、用户管理等 helper。

页面专属 helper 复用现有 `loadOnlineDataComponentScript`、`runPageLoadOnce` 和 `ensure*Ready` 模式，在进入对应页面前加载。每次只迁移一个 helper，保留明确加载错误，不以空实现伪装成功。

### 第三批：请求调度与后台任务

使用浏览器网络瀑布记录登录、罗盘、携程、美团、数据健康、收益 AI 页面：

- 合并同一页面生命周期内相同 URL 和相同参数的并发请求。
- 保留现有 TTL 和强制刷新语义。
- 页面切换时取消已经失去消费者的只读请求。
- 页面隐藏时暂停非关键轮询，恢复可见后按 TTL 判断是否刷新。
- 采集、保存和写操作不复用只读缓存，不改变完成状态。

### 第四批：CSS 与静态资源

只有在所有动态 class 被收集并具有视觉回归覆盖后，才生成项目专用 Tailwind CSS。动态 class 必须进入显式 safelist；无法证明安全的 class 不删除。

图片继续使用现有 AVIF/WebP/PNG 回退。字体和图表库继续按需加载，不为降低请求数改成更大的全局 bundle。

### 第五批：后端热路径

先记录接口总耗时、SQL 数量、最慢 SQL、返回行数和 JSON 大小，再选择优化方式：

- 重复且只读的元数据使用现有短 TTL 缓存模式。
- N+1 查询改为批量查询并按主键分组。
- 只有 `EXPLAIN` 证明扫描异常时才增加组合索引。
- OTA 采集、保存、数据库回读和状态判定不使用可能掩盖新鲜度的缓存。

## 错误处理

- 外置脚本加载失败：显示资源名称、失败阶段和刷新建议。
- 页面专属 helper 加载失败：只阻断对应页面并保留错误证据，不使整个应用退出。
- 请求取消：仅用于只读且失去消费者的请求，不显示成平台凭据失败。
- 缓存读取失败：回到真实请求并保留诊断日志，不返回假成功或旧值冒充当前数据。
- 基准工具不可用：该指标保持 `unverified`，使用静态体积和 HTTP 时序继续推进，不编造 Web Vitals。

## 测试设计

### TDD 入口保护

先增加失败测试，证明当前入口尚未满足：

- 主启动代码必须位于 `public/app-main.js`。
- `index.html` 不得继续包含 `const suxiApp = createApp(` 的完整实现。
- helper 和主脚本的 `defer` 顺序固定。
- `app-main.js` 查询版本包含当前内容哈希。
- `#app`、全局弹窗、toast 和 Vue 边界仍位于 HTML 模板内。
- 资源加载失败表面可在主脚本未执行时工作。

随后修改 `verify_public_entry_guard.mjs`，分别读取 HTML 结构与主脚本行为；现有业务 marker 检查继续覆盖组合后的应用源代码。

至少覆盖：

- 携程 EBooking。

每个页面验证可见结构、关键按钮、首个数据请求、错误表面和控制台异常。

## 性能验收

第一批必须满足：

- `index.html` 文件体积小于 2,000,000 bytes。
- 主应用脚本可以被静态 immutable 缓存，且版本哈希与内容一致。
- 重复访问时主应用脚本返回 304 或命中浏览器缓存。
- 入口、核心页面和业务合同验证无回退。

全目标完成前还必须建立并比较：

- FCP、LCP、DOM Interactive、Full Load。
- 总请求数、总传输量、JavaScript 与 CSS 传输量。
- Long Task 数量和最长任务。
- 登录和六个核心页面的接口数量与最慢请求。
- 后端核心接口 P50、P95 与 SQL 数量。

任何指标只有当前后测量环境一致时才声明改善。

## 回退策略

- 第一批回退：把 `app-main.js` 内容机械放回原 `<script>` 位置并删除引用；不需要迁移数据。
- helper 懒加载按单文件提交，每个 helper 可独立恢复同步标签。
- 请求调度优化按 endpoint key 独立关闭，不改变写接口。
- CSS 裁剪保留现有 `tailwind.min.css` 作为可恢复产物，视觉回归失败立即恢复。

## 停止条件

只有以下事实同时成立，才结束本性能目标：

1. 页面结构和全部既有功能通过约定回归。
2. 入口、核心页面、OTA、收益 AI、运营和投决链路没有功能回退。
3. 至少一组同环境浏览器前后基准证明性能改善。
4. 后端热路径已基于真实时序处理，或证据证明其不是主要瓶颈。
5. 所有未验证项已补证据或明确标记，且不存在被假成功掩盖的失败。
