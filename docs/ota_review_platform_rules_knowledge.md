# OTA各渠道点评规则知识沉淀

## 资料边界

- 来源：用户提供图片资料《各渠道点评规则详解--最新》，源文件名 `be36ca51c3b4d3b2bd6b937d2d363be9.jpg`。
- 沉淀日期：2026-06-26。
- 适用范围：OTA 点评规则、评分口径、展示口径、申诉和删除修改路径的运营知识沉淀。
- 核心目标：将点评规则服务于 `OTA数据 -> 收益分析 -> AI决策 -> 运营管理 -> 投资决策`，尤其用于口碑服务诊断、差评处理、平台分解释、转化率判断和运营动作拆解。
- 使用边界：图片本身标注为“最新”，但本次未联网或登录平台后台核验；执行删评、申诉、评分解释或对外承诺前，必须以当前平台后台、平台规则页或平台招商经理口径复核。
- 项目边界：本次只沉淀知识，不新增自动采集逻辑、不新增业务表、不改变点评默认采集策略；携程/美团点评仍不作为默认自动采集重点。

## 一、核心认知

| 结论 | 对宿析OS的意义 |
| --- | --- |
| 各平台点评的“可发表时间、有效期、追评、计分、展示”差异很大 | 点评数据不能简单按同一规则汇总；必须保留平台、来源、时间窗和计分口径 |
| 平台经常区分“展示点评”和“计分点评” | 第三方同步点评、钟点房点评、追评、低质量点评可能展示但不计分 |
| 删除、修改、申诉能力高度平台化 | 系统只能生成处理建议和证据清单，不能把“可删改”当作通用能力 |
| 近期、高质量、真实、专业用户点评通常权重更高 | AI建议应优先推动真实好评增量、差评闭环和内容质量，而不是追求机械刷量 |
| Booking、抖音、点评类平台存在独立评分和综合评分差异 | 计算模型应保留原始分制和平台综合分，不要强行用 5 分制覆盖原始评分 |

## 三、标准化字段建议

后续若进入知识库、日报或 AI 诊断，不建议先新增业务表。优先把以下字段写入知识块或 `raw_data`：

| 字段 | 含义 |
| --- | --- |
| `platform` | `ctrip`、`douyin`、`meituan`、`fliggy`、`zhixing`、`qunar`、`tongcheng`、`dianping`、`booking` |
| `rule_source` | 规则来源，例如 `user_image_20260626` |
| `rule_verified_status` | `user_provided_unverified`、`backend_verified`、`official_doc_verified` |
| `review_content_rule` | 字数、图片、视频、优质评价标准 |
| `review_open_rule` | 评价入口开放时间 |
| `review_deadline_rule` | 评价截止时间 |
| `follow_up_rule` | 是否支持追评、追评是否计分、追评截止时间 |
| `score_valid_window` | 分数统计有效期，例如 2 年、3 年、36 个月 |
| `third_party_score_mode` | 第三方点评是否只展示、是否参与计分、是否融合评分 |
| `delete_modify_rule` | 用户和商家的删除/修改能力 |
| `appeal_rule` | 申诉入口、申诉时间窗和限制 |
| `display_rule` | 展示排序、折叠、同步和第三方点评展示规则 |

## 四、系统落地规则

### 4.1 数据边界

- 这些规则属于 OTA 渠道口径，不代表全酒店经营口径。
- 点评分、点评量、差评率、回复率可进入口碑服务诊断，但不能单独推导全店入住率、ADR、RevPAR 或投资结论。
- 平台私有评分和排序权重不得反向工程；只保留可观察字段、平台展示值、影响因素和行动建议。
- 不同平台分制不同：Booking 为 1-10 分，其他平台多为 5 分或平台综合分。系统展示可做归一化，但必须保留原始分制和原始值。

### 4.2 采集与入库

- 现有项目仍优先使用 `online_daily_data`：点评类数据使用 `data_type=review`，评分可放 `comment_score` 或 `data_value`，细节放脱敏 `raw_data`。
- 不因为这份资料直接新增 `reviews` 表；只有页面、报表、预警或 AI 分析明确需要结构化读取点评明细时，才单独设计结构化字段。
- 删除、修改、申诉状态必须作为平台外部状态记录，不能因为商家侧发起申诉就把本地点评状态改为“已删除”或“已恢复”，除非有平台返回证据。
- 第三方同步点评必须标明 `source_platform` 与 `score_mode`，区分“仅展示”“融合计分”“本平台自有计分”。

### 4.3 AI 诊断使用

AI 生成点评相关建议时，必须先读取以下证据：

1. 平台与门店。
2. 点评日期、入住日期或离店日期。
3. 点评是否在有效统计期内。
4. 点评是否为自有平台点评、第三方同步点评、追评或无订单点评。
5. 是否计分、是否仅展示、是否可申诉。
6. 当前点评量、近期好评量、近期差评量和平台评分变化。

缺少上述证据时，输出 `证据不足`，不能生成确定性删评、改分或排名结论。

## 五、运营动作模板

| 场景 | 优先判断 | 建议动作 |
| --- | --- | --- |
| 携程自有点评少 | 是否少于 5 条或少于 40 条展示阈值 | 先提升真实住客自有点评数量；不要把第三方展示点评当作携程计分点评 |
| 抖音评分低 | 首次评价数量、好评率、近期好评、评价质量 | 引导真实核销用户发布图文/视频优质评价；避免低质短字评价；关注售后满意度和投诉 |
| 美团/大众点评差评影响大 | 评价时间、内容质量、用户专业度、诚信因素、近期差评数量 | 先核实事实和服务问题；能申诉的在半年内/后台处理；同时稳定新增高质量真实好评 |
| 飞猪点评展示弱 | 字数、图片数量、有效期、追评能力 | 提升图文完整度；预付订单关注 90 天评价和 180 天追评窗口 |
| 智行/去哪儿/同程评分异常 | 是否融合携程或第三方点评，第三方同步是否更新 | 拆分本平台自有点评与第三方同步点评；不要把融合分误判为单渠道分 |
| Booking 综合分变化 | 新点评权重、36 个月有效期、单项分与综合分独立 | 分析最新点评内容与综合评分，不用单项分直接反推综合分；真实问题先补救再引导客人联系平台修改 |

## 六、不可做事项

- 不把图片资料直接包装成当前官方规则。
- 不承诺任何平台一定能删评、改评或申诉成功。
- 不把追评、第三方同步点评、无订单点评和自有订单点评混算。
- 不用低质量、虚假、诱导或违规方式获取好评。
- 不在普通文档中保存订单号、住客手机号、账号、Cookie、token、投诉隐私或原始点评敏感明细。
- 不用空值、默认值或“自动修复”掩盖平台规则未核验、点评缺失或计分口径未知。

## 八、后续接入建议

1. 先将本文件作为人工知识库和 Agent 检索材料使用。
2. 若要进入系统知识库，可按 `platform + rule_type + rule_verified_status` 拆成 `knowledge_units` 与 `knowledge_chunks`。
3. 若要进入产品功能，优先做“点评规则解释卡”和“差评处理证据清单”，不要先做自动删评或自动申诉。
4. 若要进入收益分析，先把点评口碑作为影响因子，不直接把点评分转成收益结论；需要结合曝光、点击、转化、价格、库存和订单数据。
5. 若要进入 AI 决策，输出必须包含证据、平台口径、风险边界、可执行动作和复盘时间窗。
