# OTA i18n 术语与宿析OS逻辑

> 来源：用户提供的携程酒店商家后台 `i18n_translations.json`，生成日期为 2026-05-31，包含 9 个模块、6771 条词条。

## 使用边界

- 本文件用于统一宿析OS的项目描述、页面文案、字段命名和分析逻辑。
- 用户提供的携程 / Trip.com eBooking 中文前端翻译包本身不是业务数据，不代表酒店订单、流量、收益、竞争圈或服务质量事实。
- 翻译包中可能包含功能、按钮、指标、提示语、节假日、国家地区等语言资源，以及部分前端埋点上报代码；埋点代码只可作为页面行为线索，不作为经营指标来源。
- i18n 词条只能作为术语和页面语义参考；正式采集字段仍以 Request URL、Payload、Preview / Response、页面上下文和酒店/日期参数为准。
- OTA 字段默认是渠道口径，不等同于 PMS、CRS、线下直客或全渠道经营口径。
- 无法确认含义的字段保留接口字段名、source path 和证据状态，不自行改写成确定口径。

## 项目一句话

宿析OS用授权 OTA 可见数据作为经营输入，先用采集证据和字段目录固定口径，再通过质量校验和指标诊断，把收益、流量、竞争圈、服务质量、广告、商旅和运营动作接入同一条可追踪、可复盘的经营反馈链，并为投资与扩张判断沉淀证据。

## 主链路

```text
授权 OTA 可见数据
-> 采集证据
-> 字段目录
-> 标准事实
-> 收益 / 流量 / 转化 / 竞争圈 / 服务质量 / 广告 / 商旅诊断
-> 待确认 AI 建议
-> 人工确认与运营动作
-> 效果复盘
-> 经营规则与投决参考
```

## 文案规则

1. 项目描述使用“授权 OTA 数据”“经营反馈链”“采集证据”“字段口径”“动作复盘”等表达。
2. 指标展示优先使用携程/美团页面已有名称，例如“预订订单量”“预订销售额”“在店间夜”“列表页曝光量”“详情页访客量”“PSI服务质量分”。
3. 竞争圈相关文案必须写成“竞争圈平均 / 竞争圈排名 / 同圈对比”，不写成“市场平均”或“全市场排名”。
4. AI 文案必须写成“建议、解释、待确认动作”，不写成“已自动执行”。
5. P3 候选接口只写“候选、待补证据、待人工审核”，不写成“已支持”。

## 分析逻辑

| 分析对象 | 应先看 | 再判断 | 不允许 |
|---|---|---|---|
| 收益下滑 | 预订订单量、间夜量、预订销售额、平均卖价 | 是流量少、转化低、价格弱，还是库存/房态约束 | 缺分母时强行计算 ADR、RevPAR |
| 流量下滑 | 列表页曝光量、详情页访客量、订单页访客量 | 曝光问题、点击问题、下单问题分开定位 | 把访客量为 0 当作确认事实，除非接口明确返回 |
| 转化异常 | 下单转化率、成交转化率、订单提交人数 | 价格、服务质量、竞对、房型信息是否拖累 | 只用单个转化率直接给调价结论 |
| 竞争圈落后 | 竞争圈平均、排名、同时浏览率、流失去向 | 本店差距来自价格、流量、服务还是产品信息 | 不把竞争圈当作市场全量 |
| 服务质量波动 | PSI服务质量分、基础分、奖励分、减分项、回复率、点评分 | 服务质量是否影响曝光和转化 | 采集点评明文明细进入默认分析 |
| 广告投放 | 曝光、点击、花费、收益、ROAS、点击率、转化率 | 投放是否带来有效订单和间夜 | 只看花费或曝光判断效果 |
| 商旅渠道 | BPI分、商旅经营报告、协议/企业客户表现 | 与 OTA 零售渠道分开评估 | 混入普通 OTA 指标后作为统一渠道结论 |

## 落地要求

- 采集侧：保存证据、状态、失败原因和脱敏后的响应摘要。
- 存储侧：字段名、中文名、来源路径、口径、采集方式和数据状态一起入库或进入字段目录。
- 展示侧：明确“OTA渠道口径”“竞争圈口径”“商旅口径”和“待补证据”。
- 决策侧：每条 AI 建议必须绑定证据字段、影响指标、执行条件、观察期和复盘标准。
- 复盘侧：记录执行前值、目标值、执行后值、结果状态和失败原因。
