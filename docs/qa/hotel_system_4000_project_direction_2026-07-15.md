# 宿析OS 4,000 条诊断结论与项目方向

日期：2026-07-15（Asia/Shanghai）

## 结论先行

宿析OS当前最合适的方向不是继续扩成“大而全 PMS”，而是收敛为：

> **以可信携程/美团 OTA 数据为事实底座的酒店收益运营决策系统。**

项目的代码级合同、后端单测、数据质量守卫和 AI 建议边界已经较强；当前真正阻塞产品价值的不是“页面或模块还不够多”，而是三件事：

1. 真实 OTA 数据覆盖不足，部分酒店/平台/日期没有可验证载荷。
2. 真实授权采集、生产部署证明、可信代理真实 IP、次日效果和 ROI 仍依赖外部环境，不能由本地测试替代。
3. 2,688 条 P0 目标目前只有 144 条本地直接证据，且均只能保守标记为 `partial`；仍有 2,544 条 P0 证据待执行。

因此下一阶段应把 80%–90% 的精力用于“真实数据 → 指标 → 建议 → 人工审批 → 执行证据 → 效果复盘”最短闭环，而不是增加 PMS、支付、前台房态或更多展示页。

## 用例设计与真实性边界

- 基础场景：500 条，来源于软件质量、安全、可访问性、酒店消息、收益指标、AI 治理及项目业务规则。
- 组合方法：每个基础场景按权限范围、数据完整性、数据新鲜度、上游状态使用 NIST L8 两两组合，得到 4,000 条。
- 校验结果：4,000 个连续唯一 ID；4,000 个唯一语义签名；4,000 条定义校验通过；两两组合覆盖缺口 0。
- 500 条基线评估映射（仅作参考，不是变体执行结果）：通过 2,800、部分 592、失败 248、阻塞 152、不适用 208。
- 变体执行状态：`partial` 144、`not_executed` 3,856。已执行的 144 条来自六批逐例 PHPUnit 调用；没有把基线结论传播为变体通过，也没有把服务/SQLite 证据冒充完整 HTTP、UI 或真实 OTA 闭环。
- 证据类型：自动化/运行证据映射 2,720、静态/合同证据映射 904、人工/外部证据 376。

以上 4,000 条全部完成了用例结构和基线证据映射，但不等于 4,000 次真实 OTA 写入或浏览器实操。每条变体只有取得对应权限范围、数据完整性、新鲜度和上游状态组合的直接证据后，才能更新执行状态。

### 本轮长任务的 2,688 条 P0 上限

500 条基线中共有 336 条 P0；按每条 8 个 L8 组合，得到 2,688 条 P0 变体，正好落在“2,000–3,000 条”的目标区间。该目标不是重复扩容：其中 2,432 条属于本地自动化集成或合同加运行时验证，256 条必须取得授权账号、真实 OTA 载荷或外部平台证据。当前已有直接证据 144 条，剩余 2,544 条。gap-first 子集为 600 条，已有证据 64 条、待执行 536 条；外部条件未满足的场景只能保持 `blocked` 或 `not_executed`。

### Node 容量缺陷已关闭

首次执行时，`scripts/scan_p0_ota_traffic_payload_candidates.mjs` 和对应测试夹具使用同步子进程读取 P0 验证器 JSON，但没有放大默认输出缓冲。验证器有效 JSON 超过默认约 1 MiB 缓冲后被截断；测试还把 `null` 退出状态误当成成功，最终表现为 JSON 解析失败。

现已将扫描器与测试夹具缓冲提高到 16 MiB，并在解析前显式检查子进程错误和退出状态。针对性回归 5/5 通过；容量故障已关闭。后续 Node 全量出现的失败均来自前端源文件与生成产物并发写入窗口，定向校验在产物稳定后通过，与该容量问题无关。

### 运营状态机与幂等缺口已关闭

第三批先得到真实红灯：已失败/已执行任务可以被旧请求改回 `executing`；失败终态缺少回执仍会被写成 `failed`；同一缺证据的 blocked 请求重放会再次改写 `updated_at`。`OperationManagementService` 现已增加最小守卫：终态禁止倒退，`executed/failed` 都必须有证据，blocked 的同类缺证据重放返回明确冲突。第三批 24/24、465 assertions 通过，相邻运营回归 53/53、465 assertions 通过。

TC-283 审批场景仍只算 `partial`：动作窗口、诊断状态和 AI 审核状态目前由测试层业务 preflight 以及 `evidence.data_gaps` 表达，生产服务尚未把这些字段建成独立强制门禁，不能把该测试说成完整 HTTP/DB/UI 审批闭环。

### 第四批认证、AI 隐私与补偿回执缺口已关闭

第四批覆盖 TC-013、TC-270、TC-292 共 24 个 L8 组合。密码修改后会通过会话版本撤销全部旧令牌；AI 日报在调用模型前递归最小化嵌套隐私字段，并在来源不完整、核心指标缺失或数据过期时阻断模型就绪；失败动作的补偿回执会校验酒店、身份、状态、时间、影响范围和重复回执。第四批 24/24、448 assertions 通过。该证据仍不包含真实浏览器密码修改、外部 LLM 网络调用或真实 OTA 回滚，因此保持 `partial`。

### 第五批导入、点评隐私与现金流时间序列缺口已关闭

第五批覆盖 TC-200、TC-239、TC-307 共 24 个 L8 组合，并补上当日/预测周期语义边界：

- 批量导入在第 33/64 行故障时整批规范化数据回滚，原始证据和失败任务保留，移除故障后重试得到 64 条唯一记录；
- 点评姓名、格式化手机号、邮箱、证件号及嵌套标签不再进入规范化存储，失败上游不写正式快照，缺字段和过期数据显式标记；
- 投资模拟支持估值日、币种、建设期/运营期逐期现金流和期末价值，保存与回读保持同一时间序列，缺币种在持久化前拒绝；
- 当日数据误标 `historical_daily` 时纠正为分钟级 `realtime_snapshot`，未来流量预测纠正为 `next_30_days`，不破坏分钟快照桶和保存回读计数。

第五批 24/24、423 assertions 通过；相关边界 89/89、604 assertions 通过；当时 PHP 全量 1,528/1,528、15,284 assertions 通过。由于仍未覆盖完整 HTTP、页面回显和真实 OTA/投资输入，24 条证据继续标记 `partial`。

### 第七批安全与外部依赖缺口已重放

第七批覆盖 TC-104、TC-387、TC-404 共 24 个 L8 组合。TC-104 密文篡改守卫直接通过；TC-387 CORS allowlist 和 TC-404 外部依赖超时在首次执行时暴露真实红灯，完成生产修复后由 canonical runner 逐例重放通过。24 条证据均继续标记 `partial`，因为它们不代表真实生产来源、外部网络和完整用户闭环。

### 隔离浏览器 E2E 数据库边界已关闭

快速 E2E 默认使用 `hotelx_e2e` 和 `127.0.0.1:18080`。应用与辅助进程必须同时收到 `SUXI_E2E_DB_OVERRIDE=1` 和专用库名；库名只接受 `_test`、`_testing` 或 `_e2e` 后缀。连接共享当前库必须显式设置 `SUXI_E2E_ALLOW_SHARED_DB=1`，远程测试库另有独立门禁。随机前缀仍用于数据隔离和清理，但不再承担数据库隔离职责。

首次专库全量执行真实捕获到菜单模板直接读取 `window.innerWidth` 的 Vue 运行时错误；视口判断移入已暴露的脚本方法并增加契约后，4/4 用例通过。每轮清理回读均为 0，且运行器未选择、迁移或写入 `hotelx`。

该结果只证明本地隔离环境的功能与清理闭环，不替代真实携程/美团授权采集、生产部署证明、可信代理真实 IP、次日效果和 ROI 证据。

## OTA 数据闭环现状

对 2026-07-15 当前 P0 字段闭环执行只读审计：

- 平台就绪：0/2；总状态为 `blocked_by_p0_ota_gate`；
- 携程：来源行 124、流量行 62，字段事实为 `partial`；
- 美团：来源行 141、流量行 0，字段事实为 `not_loaded`；
- 匿名化酒店级候选路径 12 条（携程 10、美团 2），当前可用 0、缺失 12；
- 两个平台共同缺少 `detail_exposure`、`flow_rate`、`list_exposure`、`order_filling_num`、`order_submit_num` 的完整闭环证据。

这只是目标日期与当前验证器酒店范围的事实，不能扩大为所有酒店的长期结论。但它足以说明：**当前主要瓶颈是可验证数据供给，不是再增加分析公式。**

## 推荐产品路线

### P0：真实 OTA 事实层

- 选少量真实授权酒店，完成携程和美团同酒店、同日期的采集、保存、数据库回读和页面回显。
- 每条数据保留平台、系统酒店、平台酒店标识、业务日期、采集时间、来源方式、source path、metric key 和质量状态。
- 缺字段、权限拒绝、登录失效、过期和采集失败必须显式展示，禁止用 0、空数组或旧数据伪装成功。
- 在真实授权酒店上补齐携程与美团采集、保存、数据库回读和页面回显；隔离 E2E 数据库与本地浏览器契约已关闭，不再把本地绿灯扩大成外部完成。

### P1：收益诊断与建议可信度

- 重点做好 OTA 收入、间夜、ADR、渠道 OCC/RevPAR、取消、转化、竞品价差、广告等少数核心指标。
- 没有可售房、全渠道收入或 PMS 数据时，只能给 OTA 渠道结论，不能给全酒店经营结论。
- AI 输出必须引用输入事实、显示数据缺口和置信度；定价建议保持人工审核，不宣称已写回 OTA。
- 增加历史回放和建议后效果对比，优先证明建议是否有效，而不是继续增加提示词数量。

### P1：运营执行闭环

- 把建议转成待审批任务，记录负责人、执行时间、截图/回执、前后指标和复盘结论。
- 没有执行证据时状态只能是“建议/待执行/待验证”，不能显示“已完成”或 ROI。
- 用少量真实运营动作跑通闭环后，再扩大酒店和指标覆盖。

### 暂缓范围

- 完整 PMS 的预订、前台、房态、支付和夜审能力；
- 没有 PMS、财务、合同和产权证据支撑的全量投资尽调；
- 直接自动改价、自动投放或自动写回 OTA；
- 继续增加静态合同、页面或模块，但没有真实数据和浏览器验收。

## 方向判断

宿析OS已经不是“缺一个后台管理页面”的阶段，而是进入“证明数据与决策闭环”的阶段。最有价值的产品定位是：

> **OTA 渠道经营事实核验 + 收益诊断 + 可解释 AI 建议 + 人工可控运营执行。**

当这条链能在真实酒店连续稳定运行后，再接 PMS/财务数据，把渠道结论升级为全酒店经营和投资决策。当前立即扩成 PMS 或投资平台，会把数据证据不足的问题放大，而不会提高真实可用性。
