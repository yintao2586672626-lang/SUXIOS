# OTA 外网分析、运营 SOP 与商圈竞争脉冲知识手册

## 文档状态

- 版本：`2026-07-16`
- 用途：宿析OS知识中心检索、运营培训、诊断模板和运营任务模板参考。
- 数据级别：`external_public_reference_reviewed`。
- 适用范围：`ota_channel`。除非另有已验证的 PMS、财务或全渠道数据，不得扩大为全酒店经营事实。
- 执行边界：本手册本身不触发 OTA 改价、开关房、库存写回、活动报名或经营承诺。

## 一、OTA 公开页诊断方法

### 1. 诊断对象

诊断对象按“平台 → 维度 → 字段事实”组织。平台可包含携程、美团、飞猪等，但只有实际取得证据的平台才进入结果。

建议保留 12 个维度：

1. 平台与基础展示
2. 价格与价盘
3. 点评结构
4. 点评回复
5. 问答与咨询
6. 图片与视频
7. 房型及命名
8. 代理与分销展示
9. 未来日期价格
10. 营销活动
11. 套餐与内容展示
12. 会员或权益表达

维度只是检查目录，不代表每个平台都具备相同字段，也不代表所有字段都能从公开页稳定取得。

### 2. 字段事实合同

每个字段事实至少包含：

- `platform`：平台。
- `system_hotel_id` 与 `platform_hotel_id`：宿析OS门店与平台门店标识，不能混用。
- `business_date`、`stay_date`、`captured_at`：业务日期、入住日期和采集时间。
- `dimension`、`field_key`、`observed_value`：维度、字段键和值。
- `source_url`、`source_method`、`source_locator`：来源页面、采集方式和可复核位置。
- `evidence_ref`：截图、响应或人工核验记录的引用；不在知识文本中保存 Cookie、会员 Key 或敏感请求头。
- `quality_status`：`verified`、`observed`、`unknown`、`blocked`、`stale` 或 `conflict`。
- `confidence`：置信度只表达证据可靠性，不代替字段值。
- `saved_at` 与 `readback_status`：保存时间和保存回读结果。

### 3. 评分与未知值

- 总分只能从有证据且具有明确评分规则的字段自下而上聚合。
- `unknown`、`blocked`、`stale` 和 `conflict` 不得当作健康，也不得用零或历史值静默替代。
- 同时展示“诊断得分”和“证据覆盖率”；证据覆盖不足时，报告状态应为 `insufficient_evidence`，而不是给出看似正常的分数。
- 不设置固定起始分，不因 AI 调用失败生成伪成功报告。
- 规则权重、平台差异和适用版本必须可追溯；没有验证的权重标记为 `experimental_rule`。

### 4. 报告结构

诊断输出建议采用以下顺序：

1. 范围与证据覆盖：平台、门店、日期、采集时间、已核验字段和缺口。
2. 事实摘要：只陈述已观察事实。
3. 主要问题：问题必须引用字段证据。
4. 机会假设：明确写成假设，不把相关性包装成因果。
5. 候选动作：绑定对象、负责人、截止时间、执行证据和停止条件。
6. 30 天节奏：首周修复基础展示，第二周处理内容与口碑，第三周小范围测试价格/活动，第四周复盘；具体节奏由门店证据调整。
7. 数据缺口：列出无法判断的字段和补证方式。

### 5. 从诊断到任务

诊断问题只有满足以下合同才能转为运营任务：

- 明确平台、门店、目标页面、房型或价格计划等执行对象。
- 明确当前证据、目标状态、负责人、截止时间和审批要求。
- 执行后上传截图、平台回读或其他证据。
- 记录效果观察窗口、指标和结论。
- 未取得执行证据前，状态不得标记为完成；未取得效果证据前，不得沉淀为“验证有效的 SOP”。

## 二、OTA 运营 SOP 参考模板

### 1. 内容目录

外部静态资料可重组为 15 个知识模块：

1. 日常运营
2. 新店上线
3. 经营诊断
4. 指标口径
5. 收益管理
6. 页面设计
7. 定价
8. 促销
9. 点评管理
10. 负面舆情处理
11. 点评复盘周期
12. 绩效与协作
13. 平台差异
14. 表单与模板
15. 术语解释

原资料以章节、卡片和字段组织，并提供工作表。宿析OS应保留这种“可查、可勾选、可打印、可转任务”的结构，但不复制外站页面或运行代码。

### 2. 标准知识卡合同

每张 SOP 卡建议包含：

- `role`：适用岗位。
- `scene`：触发场景。
- `objective`：要解决的问题。
- `prerequisites`：执行前必须具备的数据、权限或审批。
- `steps`：按顺序执行的检查与动作。
- `evidence_required`：完成所需截图、平台回读或数据记录。
- `owner`、`due_at`、`review_at`：负责人、截止时间、复盘时间。
- `metrics`：观察指标及明确分母。
- `stop_or_rollback`：停止或回滚条件。
- `platform_variants`：携程、美团等平台差异。
- `source_refs`、`version`、`evidence_level`：来源、版本和证据级别。

### 3. 建议运营节奏

- 每日：检查可售、价盘、订单异常、点评/问答、活动状态和待办；只报告已核验异常。
- 每周：复盘流量漏斗、房型与价格计划、竞对变化、点评主题、动作完成率和效果证据。
- 每月：复核页面内容、图片、房型角色、促销净收益、规则阈值和已失效 SOP。
- 新店/重大改版：使用上线清单逐项核验门店身份、房型映射、基础信息、图片、价盘、库存、权益、政策和可预订链路。
- 异常事件：先保存证据与时间点，再分级、指派、处理、回读和复盘。

### 4. 指标语义修正

- “点击率”必须明确分母。可分别使用 `点击量/曝光量` 或 `详情页访客数/曝光访客数`，不能混写。
- “预订转化率”必须区分订单转化与用户转化，例如 `订单数/详情访客数`、`下单用户数/详情访客数`。
- “支付转化率”必须固定分母并写进指标定义，不能在报表之间切换。
- `OTA间夜/物理房间数`反映的是渠道销售强度，不是渠道份额；渠道份额需要全渠道间夜作为分母。
- OTA ADR 只描述对应 OTA 渠道成交价格；不得替代全酒店 ADR。
- 曝光、访问、预订、支付、入住和取消必须保留各自日期口径，不能把不同阶段混成一个转化率。

### 5. 经验阈值治理

竞对数量、回复时限、点评提及率、维度权重等静态资料中的数字只能作为候选参数：

- 标记 `reference_threshold` 或 `experimental_rule`。
- 记录来源、平台、门店类型、适用周期和版本。
- 允许门店配置，并通过执行效果复盘校准。
- 不把平台宣传、培训材料或单店经验写成平台强制规则。
- 规则过期、来源失效或效果恶化时，状态改为 `stale` 或 `rejected`。

## 三、商圈竞争脉冲方法

### 1. 观察边界

- 商圈公开页和渠道列表只能观察 OTA 渠道报价与可售状态，不能推导酒店总房态、真实剩余房量、出租率、ADR、RevPAR 或利润。
- “满房率”应改称“OTA 渠道售罄酒店占比”。只有明确观察到售罄的有效样本进入分子，`unknown`、`blocked` 和 `stale` 不进入分母。
- 外部资料宣传的 30 分钟采集频率是产品设想，不是宿析OS默认值。频率应按平台限制、账号风险、业务价值和授权范围配置。

### 2. 双时间轴

每个观察点必须同时记录：

- `captured_at`：何时看到该状态。
- `stay_date`：该状态对应哪一个入住日。

没有双时间轴，就无法区分“今天看到下周涨价”和“今天入住价发生变化”，也无法可靠计算变化事件、提前期或节假日售卖节奏。

### 3. 商圈快照

在同一平台、商圈、入住日、采集时间和可比产品口径下，可计算：

- 有效样本数、未知数、受阻数、过期数。
- 最低价、中位数、均值、最高价和价格区间。
- 明确可售酒店数与明确售罄酒店数。
- OTA 渠道售罄酒店占比。
- 本店相对中位数的价差和排名，但排名必须显示样本数与质量状态。

均值容易受极端值影响，默认应与中位数、区间和样本数一起展示。

### 4. 变化事件

连续快照应生成事件流，而不是只保留最新表格：

- `price_up` / `price_down`
- `available_to_sold_out` / `sold_out_to_available`
- `new_listing` / `listing_missing`
- `material_rank_change`
- `market_dispersion_jump`
- `holiday_pressure_change`
- `collection_blocked` / `collection_recovered`

事件必须带前值、后值、变化幅度、平台、门店、入住日、采集时间、证据引用和质量状态。

### 5. 单竞对深钻与提醒

对重点竞对，按房型或可比产品观察价格轨迹、可售/售罄切换、活动与权益、页面内容变化和点评主题变化。提醒采用“事件 → 阈值/规则 → 人工复核 → 运营任务 → 执行证据 → 效果复盘”闭环。

提醒不能直接等同于调价建议；缺少本店库存、Pickup、价格底线、转化和取消证据时，只能提示补证或人工判断。

## 四、知识状态与晋级规则

- `external_public_reference_reviewed`：公开资料已核验，可用于方法参考。
- `reference_template`：可用于创建检查表或任务草稿，尚未验证效果。
- `experimental_rule`：候选阈值或规则，必须在门店和周期内试验。
- `validated_sop`：任务已执行，具备执行前后证据与效果复盘，且适用边界明确。
- `stale`：来源、平台版本或适用条件过期。
- `rejected`：实践证据表明规则无效或风险不可接受。

只有 `validated_sop` 才能在相同适用条件下作为优先建议；即使如此，仍需当前门店数据和人工复核。
