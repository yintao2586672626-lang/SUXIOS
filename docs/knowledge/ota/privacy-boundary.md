# OTA 隐私、安全与知识边界规范

## 1. 核心原则

宿析OS学习的是验证方法、数据流程、状态机和设计思想，不是复制某个网站。任何外部系统材料都只能作为能力设计输入，不能直接成为代码、接口、凭证或客户数据。

优先级：

~~~text
授权边界
-> 最小必要数据
-> 脱敏与可追溯
-> 明确质量状态
-> 可审计访问
-> 到期删除
~~~

## 2. 禁止进入宿析OS知识或普通运行日志的内容

| 类别 | 禁止内容 |
| --- | --- |
| 身份凭证 | 明文账号、密码、短信验证码、认证密钥 |
| 会话材料 | Cookie值、Authorization、访问令牌、刷新令牌、完整浏览器会话 |
| 原系统实现 | 整套源码、私有路由、真实接口地址、页面脚本、品牌组件 |
| 平台敏感数据 | 未经授权的订单明细、客户联系人、手机号、姓名、证件和支付信息 |
| 真实运行材料 | 未脱敏请求头、完整响应、截图中的二维码或会话信息 |
| 识别性标识 | 可直接用于访问外部平台的账号、Profile路径或设备标识 |

文档中可以出现“密码、Cookie、令牌、接口地址”等边界词，用于说明禁止项；不得出现真实值、可复用片段或完整访问路径。

## 3. 允许保存的能力证据

允许保存：

- 宿析OS内部的酒店引用、平台代码和受控POI引用；
- 会话是否已验证、验证时间和失败原因；
- 绑定、权限、数据探针和采集任务的状态；
- 脱敏后的来源引用、抽象字段路径和标准化指标；
- 合成 fixture、虚构日期和虚构数值；
- 访问审计记录、证据摘要和质量状态。

如果必须关联外部对象，只保存不可直接访问平台的受控引用，并由权限系统保护。不要把引用设计成可重放的凭证。

## 4. 采集与知识沉淀边界

| 层级 | 可以沉淀什么 | 不可以沉淀什么 |
| --- | --- | --- |
| 能力规范 | 验证顺序、字段合同、状态机、失败分类 | 原系统页面和路由复制 |
| 脱敏证据 | 最小字段、抽象来源路径、采集时间 | 完整原始响应和敏感请求材料 |
| 标准事实 | 指标值、口径、范围、质量状态 | 缺失时的伪造值或不明来源值 |
| 业务结论 | 有证据的 OTA 诊断和动作 | 将OTA数据包装为全酒店事实 |
| 测试材料 | 合成 fixture 和失败用例 | 真实账号、客户数据和真实令牌 |

用户提供的材料应标记为 user_provided_unverified，除非已经在宿析OS自己的环境中重新验证。平台规则、法律、价格、权限等时效性事实不能仅凭旧材料视为当前官方规则。

浏览器采集只能使用账号所有者已登录的受控 Browser Profile。适配器不得把已存 Cookie、令牌或其他会话材料写入临时文件、命令行参数或浏览器上下文；Profile 未准备或登录失效时必须返回 `waiting_config`，由人工完成登录后再验证。

## 5. 数据生命周期

1. 接收：先确认授权范围和用途；
2. 采集：仅采集目标酒店、POI、日期和能力所需的最小字段；
3. 脱敏：在进入普通日志、知识库或导出包前去除凭证和客户识别信息；
4. 标准化：保留来源引用、数据日期、采集时间和质量状态；
5. 使用：按 OTA_CHANNEL_SCOPE 或 WHOLE_HOTEL_SCOPE 限制下游；
6. 复核：对失败、过期、未验证和部分数据保留可排查线索；
7. 删除：按授权和保留策略删除临时材料、过期证据和无用原文。

任何“为了方便调试而长期保留完整会话或完整响应”的做法都不符合本规范。

## 6. 访问与审计要求

- 读取平台数据、查看敏感能力和导出证据必须经过权限检查；
- 订单、评价和客户相关能力应按最小字段、最小人员和最小时间授权；
- 审计记录只保存操作者、酒店、平台、能力、时间、结果和内部证据引用；
- 普通日志不得打印请求头、Cookie、令牌或完整客户字段；
- 失败信息应足够定位问题，但不能通过错误信息泄露凭证；
- AI上下文只接收已验证的标准事实、质量状态、证据引用和范围标签。

### 6.1 同步任务、日志、资源目录与数据源状态

- 同步任务的 `message`、`sync_diagnostics`、任务统计和同步日志只能保存状态码、日期、计数、字段状态、P0状态、缺口代码和下一步动作；不得保存适配器返回的错误原文。
- `adapter_message`、完整异常文本、请求头、Cookie片段、Authorization片段和令牌片段不得进入任务统计、日志、状态接口或页面提示。
- 数据源列表只能返回 `has_secret`、`has_cookies` 等不透明存在性指示；不得返回 Cookie 预览、凭证前后缀或可重放片段。
- 资源目录的 `missing_reason` 与 `latest_task.message` 同样只能返回允许的原因码；即使读取的是历史任务或数据源记录，也不得回显旧的外部错误文本。
- 历史记录在读取时也必须再次走同一白名单，不能因为旧数据已经落库就绕过当前隐私边界。

## 7. 安全验收用例

| 检查 | 预期结果 |
| --- | --- |
| 扫描六份文档中的敏感模式 | 无真实凭证、令牌、Cookie值、Authorization值或客户隐私 |
| 扫描外部系统路径 | 无可直接复用的真实接口地址或私有路由 |
| 检查测试 fixture | 仅含合成数据和虚构引用 |
| 检查日志契约 | 只记录脱敏证据引用，不记录完整请求材料 |
| 检查OTA范围 | 每个OTA事实有 metric_scope=OTA_CHANNEL_SCOPE 或明确范围 |
| 检查全酒店结论 | 没有完整 PMS/财务/全渠道证据时被阻断 |
| 检查失败显示 | 不使用空数组、0或旧数据伪装成功 |

## 8. 文档自检清单

- [ ] 没有明文账号、密码、Cookie、令牌或Authorization；
- [ ] 没有客户手机号、姓名、证件或完整订单明细；
- [ ] 没有原系统完整源码、真实接口地址或私有路由；
- [ ] 没有把外部页面数字写成宿析OS测试真值；
- [ ] 没有把OTA渠道数据描述成全酒店事实；
- [ ] 所有失败、缺失、过期和未验证状态都可见；
- [ ] 所有可用指标都具备来源、时间、口径和范围；
- [ ] 所有示例都可以在没有真实凭证的环境中理解和测试。
