# Revenue AI 核心范围优先级

更新日期：2026-07-17

本文件用于把“只保留核心功能”的产品边界固化为当前重构基线。它不是删除旧模块的执行命令；任何删除、隐藏或迁移仍必须按最小改动、旧链接兼容和验证结果逐项执行。

## 主产品边界

主产品名称：宿析OS 收益管理智能体 Revenue AI。

核心使用者包括酒店行业专家、多店老板和培训老师。系统优先输出可核验的数据结果、对比结果与异常信号；AI提供辅助解读，不承诺替代用户完成专业分析或最终判断。

核心日常链路：

```text
携程/美团昨日 OTA 数据
-> 异常判断
-> 竞对对比
-> 动态定价建议
-> 今日执行动作
-> 明日效果复盘
```

第一版只解决早晨高频问题：昨天 OTA 表现如何、数据是否可信、哪些数据缺失、是否具备调价判断条件、今天谁去执行、明天如何用真实证据复盘。

## P0 保留

| 模块 | 当前入口 | 保留原因 | 边界 |
|---|---|---|---|
| Revenue AI 总览 | `收益管理智能体总览` / `compass` | 每日收益管理主入口 | 读取 `/api/revenue-ai/overview`，不伪造成功，不直接写 OTA |

## 新增待完成清单：专业检测经验吸纳

以下事项只吸收医院检测设备在数据质量、结果分层和复查逻辑上的经验，不把宿析OS改造成医疗系统，也不引入“挂号、病例、药方”等产品命名。

| 优先级 | 待完成项 | 最小验收标准 |
|---|---|---|
| P1 | 统一结果分层契约 | API、报告和页面能明确区分来源事实、派生指标、异常信号、AI辅助解读和人工判断；缺失项保持显式，不混写为单一“AI结论”。 |
| P1 | 为异常信号补齐参考依据 | 每条异常标明历史、同期、竞品或其他同口径参考；没有合适参考系时只展示测得值，不强行判断正常或异常。 |
| P1 | 分开报告可用状态与经营闭环状态 | 已验证的数据结果可以在AI失败、尚无建议或尚无ROI时保持可用；建议、审批、执行和复盘使用独立状态表达。 |
| P2 | 收敛AI辅助判读格式 | AI优先返回可能解释、冲突证据、缺失信息和置信程度；不把相关性写成确定原因，不直接替代专家或老板判断。 |
| P2 | 建立同口径后续观察约束 | 执行前后比较尽量保持酒店、平台、日期基础、指标定义和参考范围一致，并继续声明前后变化不单独证明因果。 |
| P2 | 验证多角色结果呈现 | 在不复制三套业务逻辑的前提下，验证老板摘要、专家证据视图和匿名培训案例对同一事实结果的不同呈现；培训视图不得默认访问真实跨酒店数据。 |
| P1 | 固化单项分析结果契约 | 每种结果明确分析对象、来源、业务日期、采集时间、指标口径、参考依据、刷新或有效期、缺失状态和适用边界；分析方法留在后台，不要求用户阅读过程才能理解结果。 |
| P1 | 补确定性结果版本标识 | 相同输入、指标版本和参考版本应产生相同确定性结果；报告可追溯结果版本，AI文本不得改写已计算事实和指标。 |
| P1 | 统一对外边界表达 | 产品页、试用说明、系统页面和导出报告一致说明：OTA信号不等于全酒店结论，价格或满房变化不单独证明需求，AI不替代老板或专家判断。 |
| P2 | 用真实授权数据验证首次价值 | 真实采集稳定后，用客户自己的授权酒店和所选竞对完成首次可信采集及至少一次同口径对比；演示数据继续显式标注，不把AI结论或ROI承诺设为试用成功条件。 |
| P2 | 建立异常优先级与降噪控制 | 多店老板和专家可选择查看全部变化或重点异常；优先级、阈值和未提醒原因可追溯，不用静默过滤隐藏数据缺口。 |
| P1 | 建立参考集治理机制 | 每份对比结果记录竞对/商圈/历史窗口的选择人、选择时间、适用门店、有效期和可比性说明；参考集变化后新旧报告不能直接混比，必须标明参考版本。 |
| P1 | 记录专家校正与人工结论 | 专家或老板可以对AI解释、异常等级和参考集提出接受、驳回或修正意见；系统保留校正原因、操作者和时间，用于复盘与评估，不自动把单次人工意见扩散到其他酒店。 |
| P2 | 固化可分享结果交付件 | 面向老板、专家和培训老师输出同一事实底座的摘要页、证据页和脱敏案例页；导出件必须带来源、日期、口径、缺失项和AI辅助边界，避免只截图或只复制聊天文本。 |
| P2 | 建立试用验收指标 | 试用是否成功优先看首次可信采集成功率、第二次同口径对比完成率、关键缺失项暴露率和用户是否确认结果有用；不以页面访问量、AI生成字数或承诺ROI作为核心验收。 |

### 竞品证据与采用边界

2026-07-16 复核[商圈全局眼](https://eye-intro.fjhoteltools.cn/)公开页面：其一天体验要求使用客户自己的商圈和竞对完成首次采集及至少一次后续对比；页面同时明确价格和满房只是市场参考信号，不构成需求爆发证明或自动调价建议。宿析OS只吸收“真实数据验证首次价值、购买前讲清边界、重点信号降噪”三类经验。

该产品当前属于邻近的市场信号工具和入口替代风险，不按宿析OS的完整目标用户与能力范围视为直接同类产品。不得因为竞品存在而复制其单店运营定位、按监控酒店数定价、人工支付流程或页面功能清单；宿析路线仍由行业专家、多店老板和培训老师的真实问题主导。

## P2 下沉

这些模块保留旧 path 和旧能力，但不再挤占主工作台；统一挂在“全生命周期辅助”下，默认不作为 Revenue AI 首页闭环的一部分：

- 投资、筹建、战略推演、量化模拟、可行性报告
- 开业准备、开业检查
- 扩张、市场评估、标杆选模、协同提效
- 转让、资产定价、时机推演、转让数据看板
- 图片优化助手
- 酒店 AI 工具箱

## 不做

- 不重写整个系统。
- 不删除旧模块、旧 path 或历史数据。
- 不新增生产依赖。
- 不改现有核心业务表字段；只允许为结果版本和人工评审新增独立、追加式表，并保持旧报告兼容。
- 不把 OTA 渠道口径包装成全酒店经营口径。
- 不自动写携程/美团价格。
- 不用空数据、默认值或宽泛 catch 掩盖采集失败、登录失效、缺字段或接口未命中。

## 验收线

自动验收由 `npm.cmd run verify:core-scope` 覆盖，至少检查：

- 一级导航只有 `收益管理 / 线上数据 / 运营执行 / 系统设置`。
- 默认首页内部 path 仍是 `compass`，用户可见名称是“收益管理智能体总览”。
- “全生命周期辅助”位于收益管理二级入口内，旧二期模块只下沉不删除。
- 普通经理不可见 `agent-center` 超级管理员工具箱。
- Revenue AI 首页使用 `/api/revenue-ai/overview` 和 `public/revenue-ai-static.js`。
- `ota_room_revenue`、`ota_room_nights`、`ota_adr`、`ota_contribution_revpar` 继续显式区分 OTA 渠道口径与全酒店口径；缺全酒店可售房晚时保留 `available_room_nights_missing`。
- 调价建议只能走人工审核、转运营执行意图、执行证据和 ROI 复盘证据，不直接写 OTA。
