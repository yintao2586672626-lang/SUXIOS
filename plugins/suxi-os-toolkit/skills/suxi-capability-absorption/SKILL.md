# 宿析能力吸收与复刻

## 核心契约

- 把“学”解释为理解材料后立即用于当前任务，不把摘要、能力卡、安装记录或实施计划当作最终结果。
- 把“复刻”解释为复现对用户有价值的输入、动作、输出和失败状态，不盲目复制不兼容代码或臆造网站隐藏后台。
- 用户主动提供的材料默认有学习价值。主动结合当前语境、宿析OS现有入口和产品链识别一个最高价值点，不反问用户“这里有什么可学”，也不要求用户先整理借鉴清单。
- 默认选择一条最接近当前目标的实施主线，不把“有学习价值”扩大为全量接入；只有两个以上方向同样合理且会接入不同业务模块、改变数据口径或写入范围时，才问一个决定性问题。
- 默认使用交付模式。只有用户明确说“只学习”“只评估”“不要改代码”时，才使用 `learn_only`。
- 先读项目 `AGENTS.md`，检查当前工作区和相关代码，保护用户已有改动。
- 只处理用户给出的材料与目标功能，不扩成全项目审计、架构重写或发布治理。

## 统一入口

用户可以直接给出文件、目录、仓库、Skill、Prompt、接口文档、MCP、SQL或数据文件、网页链接、应用、截图/Figma、录屏、SOP/PRD、日志、测试、用户示教、版本差异或成品样例，无需填写技术表单。内部先整理一张最小接入卡：

```text
材料类型 / 材料位置与来源指纹 / 授权范围 / 目标用户结果 /
现有入口 / 可验证事实 / 推断与未知 / 复用方式 / 黄金样例 /
预计修改面 / 验收方式 / 停止条件
```

目标结果能从当前上下文确定时直接推进。用户只发送材料或只说“这个给你”时，从当前任务、最近明确对象、材料行为和宿析OS现有入口中推断一个最佳落点。只有目标功能、授权范围或真实写入边界不确定且会导致错误实现时，才问一个决定性问题。

## 材料路由

保留基础入口 `Skill / 源码 / 网站 / SOP`，并扩展为 `Skill / 源码 / API / 网站或应用 / 数据 / SOP / 行为证据 / 版本与成品样例` 八类路线；每类都可以组合截图、录屏、日志、测试或用户纠错作为补强证据。

确定材料类型后读取 [source-routes.md](references/source-routes.md) 中对应章节；需要判断学习深度、证据强度、纠错或防遗忘时读取 [learning-protocol.md](references/learning-protocol.md)；材料来自互联网且包含 Skill、插件、Prompt、脚本或智能体配置时，必须读取 [online-skill-practices.md](references/online-skill-practices.md)。只启用真正需要的项目 Skill 或工具。

- Skill / Prompt / 智能体 / 自动化：核对规则、工具、脚本、资产和依赖；支持 n8n、Dify、Coze 等 JSON/YAML 工作流导出；在线来源先做静态预览、来源锁定和权限审查，再从目标业务入口在当前样例上真实使用一次。
- 源码 / 仓库 / SDK / CLI：运行真实入口，抽取行为合同，再选择复用、适配或原生重建。
- API / OpenAPI / Postman / MCP / Webhook：验证真实请求与错误响应，完成鉴权边界、字段映射、调用和失败回显。
- 网站 / 应用 / 截图 / Figma / 录屏：从可见行为和用户示教提取任务路径，不臆造隐藏后台。
- SQL / CSV / Excel / JSON / XML / 数据导出：验证结构、语义、范围和样例，继续闭合保存与回读。
- SOP / PRD / 手册 / 规范：抽取角色、状态、审批、证据和异常，把文档变成可操作流程。
- 日志 / 报错 / 测试 / 工单 / 用户反馈：从失败和纠错中提取规则，并用回归样例证明已修正。
- Commit / Diff / Changelog / 模板 / 示例成品：识别真正变化和输出标准，不机械复制历史或外观。

## 学习强度协议

不用“学了七七八八”描述状态，只使用以下可证明阶段：

1. `observed`：材料已打开或能力已实际运行，事实、推断和未知已分开。
2. `understood`：输入、动作、输出、状态、副作用、权限和失败合同已经明确。
3. `reproduced`：至少一个真实样例可以稳定重放；有条件时同时覆盖正常、边界和失败样例。
4. `integrated`：能力已进入宿析OS真实入口，核心动作、保存/产出、回读和失败状态可验证。
5. `guarded`：来源指纹、实际加载副本、黄金样例、基线对照、回归测试或 verifier 已沉淀，来源升级或用户纠错后能定向复验和回滚。

`learn_only` 最多报告到 `understood`；功能任务至少达到 `integrated` 才能标记 `complete`；反复使用、插件分发或关键数据链能力应达到 `guarded`。

## 闭环凭证

每个目标能力都必须形成一条可核对凭证：

```text
来源事实 -> 宿析OS指定入口 -> 实际动作 -> 可观察产出 ->
保存与回读（或明确不适用） -> 关键失败/缺失路径 -> 验证结果
```

- 新建独立 Demo、复制文件、保存 Prompt、安装 Skill、导入 SQL、生成静态页面或成功提示，都不能替代“指定业务入口已经调用新能力”。
- 交付模式至少实测一个代表性成功样例和一个关键失败、缺失或未验证样例；不能用 mock、硬编码或默认值补齐证据。
- `complete` 由所需凭证全部通过得出；已有可运行结果但缺任一必要凭证时为 `partial`；只有不可由当前权限或材料替代的外部阻塞才为 `blocked`。
- 外部材料进入 OTA、收益或 AI 链路前，必须验证平台、系统酒店、平台门店或绑定、目标日期、来源方法、采集/导出时间和质量状态；关键身份缺失时不得向下游放行。

## 执行顺序

1. **取得证据**：确认材料真实存在，记录版本、日期、路径或哈希等来源指纹，读取入口与关键依赖；区分源码事实、可见行为、用户目标、推断和未知。
2. **预览信任**：在线 Skill 或插件在安装、执行脚本、声明工具权限前，运行 `node scripts/inspect-skill-package.mjs <本地Skill目录>`，人工复核完整文件树、风险项、许可、兼容性、依赖、同名冲突与来源提交。静态检查不能证明安全。
3. **选择路径**：按 `直接复用 -> 适配接入 -> 行为复刻 -> 最小重建` 选择最短可验证方案。
4. **建立样例**：优先取得一个正常样例、一个边界样例和一个失败样例；材料不足时显式标记缺项，不编造样例。
5. **进入实现**：完成一次有边界的分析后立即转入实现；除非测试失败指回材料，不重复从头扫描。小任务读取项目规则、材料入口和目标入口后，下一步必须写入实现或报告真实阻塞；连续三轮定向读取仍没有新增决定性证据时必须转入实现、缩小范围或报告唯一阻塞。
6. **闭合功能**：优先形成 `用户入口 -> 主要动作 -> 保存/产出 -> 回显/读取 -> 真实失败状态`。
7. **重放与基线**：用原始样例或黄金样例重放输入，比较输出、状态、副作用和错误；Skill质量变更还要在干净上下文中对比新版与旧版或无Skill基线。涉及页面或系统交互时打开实际页面确认。
8. **沉淀复用**：功能闭环后，再把稳定合同、用户纠错、黄金样例补入项目 Skill、reference、eval 或 verifier，并保留来源指纹。不得用“新建了 Skill”替代业务功能交付。

## 学不会时的复刻策略

当原材料无法直接运行、技术栈不兼容或细节不完整时，不继续无限研究：

1. 提取稳定输入、输出、状态和用户路径。
2. 能适配就写薄适配层，不强行迁移整套框架。
3. 不能适配就按可见行为在宿析OS原生重建。
4. 隐藏后台、私有接口或缺失算法不能猜测；先交付不依赖它们的最小可用部分。
5. 只有缺少授权、账号、必要文件、真实数据或不可替代外部能力时才标记 `blocked`，并写清唯一下一步。

## 完成门禁

满足以下条件才能标记 `complete`：

- 目标入口已经存在，使用者能够找到；
- 新能力已经从用户指定的宿析OS模块或业务入口真实调用，而不是只在隔离 Demo 中运行；
- 主要动作能够真实执行；
- 需要持久化时已验证保存与回读；
- 缺失、失败和未验证状态没有被默认值或话术掩盖；
- 最小相关验证通过；Skill 接入类任务还必须完成一次真实触发或样例使用。
- 已报告当前学习阶段；功能交付不得低于 `integrated`，需要长期复用时不得缺少 `guarded` 证据。
- 在线Skill已记录来源仓库、版本或提交、文件树指纹、许可/兼容性、依赖和权限审查结果；同名遮蔽没有被静默忽略。
- 可分发或升级的Skill已核对项目权威版、插件分发版和本机实际加载副本；复杂升级有旧版或无Skill基线与可回滚版本。

以下情况一律不是完成：

- 只总结了材料；
- 只列了可借鉴点；
- 只创建能力卡、方案、待办或 Skill；
- 只复制源码、导入 SQL 或还原页面外观；
- 只下载、安装、结构校验或静态扫描了在线Skill；
- 只说“理论上可以”，没有运行、保存、回显或验证证据。

未闭合时使用明确状态：

- `partial`：已产生可运行结果，但仍缺一项验收；
- `blocked`：存在无法由当前权限或材料解决的外部阻塞；
- `learn_only`：用户明确要求只学习或只评估。

## 输出

按项目代码任务格式，只报告：改动范围、核心修改、验证结果、风险点。先给真实结果；不要用长篇学习笔记替代交付。
